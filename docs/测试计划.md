# KubeXM 全参数覆盖深度排期与环境测试计划 (Exhaustive Test Plan)

## 1. 计划宗旨
本测试计划设计为为期 **3周** 的大排期，将深度贯彻测试方案的原则，系统性地对 `kubexm` 所有指令、子指令及 30+ 项复杂参数（包含 `--dual`, `--parallel`, `--output-format`, `--with-build-*` 等）进行沙盘推演和上机实操验证。

## 2. 三维测试拓扑与环境矩阵 (Test Environments)
为了确保各个层级的适配性和隔离状态测试到位，预备了以下三大集群测试池：

| 拓扑标识 | 网络层级 | K8s架构配置 | 验证重点组件域 (internal/step) | 核心参数覆盖面 |
| --- | --- | --- | --- | --- |
| **ENV-A** | 纯公网高速 | containerd + Cilium | `download`, `iso`, `cni/cilium_delete` | `--with-build-os`, `--cni`, `--output-format` |
| **ENV-B** | 物理 Air-Gap (断网隔离) | docker + Calico | `registry`, `images/packages_concurrent`, `kubeadm`, `cni/calico_delete` | `--parallel`, `--dual`, `--manifest`, `--target-registry`, `--packages-dir` |
| **ENV-C** | 异常干扰压测环境 | crio + Flannel | `certs/kubeadm`, `certs/etcd`, `etcd`, `cni/flannel_delete` | `--cacert`, `--cert`, `--key`, `--endpoints`, `--force`, `--backup`, `--path` |

## 3. 分阶段深度排期流转 (Execution Schedule)

### 阶段一：防御机制拦截与静态渲染排查 (Days 1 - 2)
**目标**：测透 `manifests` 静态渲染及底层 YAML Schema 的零容忍拦截。
- **配置免疫测试**：输入不受支持的操作系统、组件版本甚至破坏 YAML 格式的字符串，检查程序能否在发往底层执行器前瞬间安全截断。
- **静态资源推演**：使用组合指令 `kubexm create manifests --kubernetes-version=v1.34.2 --kubernetes-type=kubexm/kubeadm --cni=cilium --arch=arm64` 并配合 `--output-format=json`，在控制台捕获并校验 JSON 结构输出。

### 阶段二：全域资产打包与跨架编译 (Days 3 - 5)
**目标**：测试 `download` 的依赖管理及 `create iso` 的容器化交叉编译链。
- **参数强覆盖**：通过追加 `--kubernetes-version=v1.27.2 --container-runtime=docker` 强制接管 `config.yaml` 属性，校验生成的 `*.bom.txt` 与缓存包绝对匹配。
- **多端打包**：轮替输入所有参数，包括 `--with-build-os=ubuntu22`, `--with-build-os-version=22.04`, `--with-build-arch=amd64`, `--with-build-all`, `--with-build-local`，验证在 ENV-B 中可以利用该产物顺畅自举。

### 阶段三：Registry 私库高频吞吐与双标推图 (Days 6 - 7)
**目标**：全面覆盖 Registry 及内置推图客户端。
- **一键生杀**：测试 `create registry` 以及附带 `--force` (强制绕过交互) 和 `--delete-images` 联合使用的 `delete registry` 清洁能力。
- **高并发推流流转**：执行 `push images --packages --packages-dir=xxx --parallel=20 --list=my-images.txt --dual`。极限界压测其并发队列、双重打标系统（保留原生与附加内网 registry 前缀），并结合 `--manifest` 与 `--target-registry=hub.remote` 测试 OCI 跨架构清单聚合。

### 阶段四：核心生成机制与节点漂移驱逐 (Days 8 - 12)
**目标**：测透在线/离线的双管部署及弹性安全伸缩 (`scale cluster`, `delete cluster`)。
- **隔离度测试**：在 ENV-B 中断网下使用预缓存包和本地私有库全拉起集群，坚决不允许抛出连接公网域名的异常。
- **安全对接测试**：对接外部环境时引入 `--endpoints`, `--cacert`, `--cert`, `--key` 开展双向 TLS 通信。
- **业务驱逐保护机制**：核心测试 `scale cluster --action=scale-in --role=worker --nodes=xyz`。必须捕获到工具向 K8s 发起 `drain` 调用，证明节点退线前的业务安全漂移策略有效。
- **备份优先级验证**：针对带有 `--backup` 参数的 `delete cluster --force`，断言核心代码逻辑必先对数据落盘，方可继续破坏执行。

### 阶段五：组件热更及内部证书重颁发 (Days 13 - 16)
**目标**：探查内部组件（如 `certs/` 及运维体系命令）交互。
- **升级轮转**：带 `--to-version` 执行 `upgrade cluster/etcd`，观测滚动替代及业务防跌断效果。
- **证书网格下发**：逐个执行 `renew kubernetes-ca / etcd-ca / kubernetes-certs / etcd-certs`。验证从 `rotate_collect` 至 `distribute_pki_k8s_copy.sh` 等底层脚本能否确保将新签证书妥善挂载，并使对应服务热重载。
- **探测与复原**：执行 `reconfigure cluster --target=runtime/etcd/cni/lb/addons/hosts` 进行细粒度更新；执行 `health cluster --check=node/component/workload/all` 打印诊断表；以 `--path=/xx.db` 为参数测试 `restore cluster`。

### 阶段六：架构底线测试与发版回归 (Days 17 - 18)
- 任何处于部署当中的 Pipeline 阶段，人工拔掉网线或屏蔽 22 端口，观测 Connector 如何依照预定规范输出“Timeout”重试并安全中止。清理回归所有用例后给出签发报告。
