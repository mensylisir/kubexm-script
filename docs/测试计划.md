# KubeXM 全参数覆盖极限界测试计划 (Exhaustive Test Plan)

## 1. 计划目标
基于最高要求的全矩阵测试策略，本测试计划设计为期 **3周**。任务不仅是把主线跑通，而是必须验证命令行中解析出的多达 30+ 项的每一个 flag（包括 `--manifest`, `--target-registry`, `--cacert`, `--output-format` 等冷门高级参数）并确保组合下在不同网络切面（纯公网/Air-Gap）表现稳固，严格落实防御架构。

## 2. 三维测试拓扑矩阵 (Test Environments)
分配以下环境群集以供全量参数注入演练：
| 拓扑标识 | 网络层级 | 操作系统及版本 | K8s 运行态组件 | 核心侧重测试路径 |
| --- | --- | --- | --- | --- |
| **ENV-A** | 纯公网高速 | Ubuntu 22.04 + Debian 11 | containerd + Cilium + Kube-VIP | 在线全要素下载、`--output-format` 输出检查、单节点创建。 |
| **ENV-B** | 物理断网隔离 (Air-Gap) | Rocky Linux 9 + CentOS 8 | docker + Calico + HAProxy | 纯离线安装包（Packages）的分发、本地 Registry 拉起及 `--dual` / `--manifest` 并发推图，节点驱逐。 |
| **ENV-C** | 异常干扰环境 | CentOS 7.9 | crio / podman + Flannel | 外部鉴权测试（`--cacert`, `--endpoints`）、Connector 连接中断/超时重试压测、灾备降级。 |

## 3. 分阶段深度执行排期

### 阶段一：防御拦截与解析边缘压测 (Days 1 - 2)
**目标**：测透 Config Schema 防守机制与 `manifests` 静态推演。
- **配置免疫攻击**：投递格式错乱、包含未注册字符、类型倒置的 YAML 文件，检查 `internal/config/validator` 是否能在 0.1 秒内精准驳回，不允许进行任何 SSH 环境接触。
- **清单演算穷举**：利用嵌套循环执行 `kubexm create manifests`，交替插入全集参数 `--kubernetes-version=v1.x`, `--kubernetes-type=kubexm/kubeadm`, `--cni`, `--arch=amd64/arm64`，输出结果比对断言。

### 阶段二：全资产隔离编译与精准下发 (Days 3 - 5)
**目标**：测试 `download` 及 `create iso`。
- **强制属性覆盖**：执行 `kubexm download` 时追加 `--kubernetes-version`、`--container-runtime` 等参数，测试其取代 YAML 内定属性的覆盖度，并校验四种 `*.bom.txt` 清单的 SHA256 同源性。
- **ISO 跨界打包**：对 `create iso` 命令使用所有相关参数 `--with-build-os`, `--with-build-os-version`, `--with-build-arch`, `--with-build-all`, `--with-build-local`，产生多份镜像并于 ENV-B 虚拟机内执行 ISO 原生挂载校验。

### 阶段三：私有 Registry 架构与并发路由流转 (Days 6 - 7)
**目标**：测试内网 Registry 及 `push images` 极限界传参。
- **一键生灭**：执行 `create registry`，再使用等价于强行按钮的 `delete registry --force --delete-images` 验证不留痕迹。
- **复杂镜像分发**：执行 `push images`，同时传入 `--packages-dir`, `--list`, `--parallel=20`, `--dual`。
- **多架构清单注入**：额外带上 `--manifest`（自动生成 OCI 多架构清单记录）及 `--target-registry` 指定另一异地远程库。观察高并发管道下的 OOM 或日志阻塞情况。

### 阶段四：全域生命周期控制与高可用集群生成 (Days 8 - 12)
**目标**：测试核心的 `create cluster`, `delete cluster` 及 `scale cluster`。
- **环境对撞部署**：在 ENV-A 跑通极致的公网极速下载安装；在 ENV-B 切断一切路由下，依靠 local registry 完成离线拓扑。
- **外部 API / TLS 对接**：在连接外部存量数据库或外部 LB 时，使用 `--endpoints`, `--cacert`, `--cert`, `--key` 附加于安装及对接指令进行双向 TLS 校验。
- **危险绕过机制**：带 `--force` 执行 `delete`。附加 `--backup=/dir/x.db` 确保灾备在销毁前强制生效。
- **规模伸缩驱逐机制**：通过参数 `--action`, `--role`, `--nodes` 控制伸缩。严测 `--action=scale-in`，必须捕获 K8s API 执行 `drain`（相当于后台的优雅退线按钮），等待 Pods 驱离方可拆包。

### 阶段五：集群无缝诊断与底盘维修 (Days 13 - 16)
**目标**：测试 `health`, `reconfigure`, `upgrade`, `renew`, `backup/restore` 运维体系。
- **输出格式及深层探针**：使用 `health cluster --check=node/component/workload/all --output-format=json/table` 验证多路探针提取精准度及终端兼容。
- **单点配置热下发**：执行 `reconfigure cluster --target=runtime/etcd/cni/lb/addons/hosts` 检查针对单一服务的 `systemctl reload` 机制。
- **无感轮转**：`upgrade` 时利用 `--to-version` 拦截错误跨度；`renew` 执行根及叶子级 (`-ca` / `-certs`) 的证书轮调并观测自动刷新。
- **数据恢复闭环**：以 `--path` 定位快照进行 etcd 的 `backup` 和 `restore`。

### 阶段六：容错性架构破坏测试与验收 (Days 17 - 18)
- 在执行上述各类流水线任意步骤时，人工操作 Linux 底层切断网络或 `kill` SSH 进程。要求引擎层必须按照超时限额自愈，并在最后彻底断连时，按规范打印 `[Task]->[Step]` 层级日志，杜绝底层 bash 的无限期挂死。完成回归评估并输出放行报告。
