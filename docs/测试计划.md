# KubeXM 全面测试计划 (Test Plan)

## 1. 计划概述与目标
本测试计划定义了 KubeXM 测试任务的具体执行阶段、时间线、所需资源配置以及交付物要求。测试执行采用阶段渐进式方法，从基础模块的冒烟测试逐步过渡到复杂分布式架构的系统级演练，确保每一次迭代的质量。

## 2. 团队角色与职责 (Roles and Responsibilities)
- **测试负责人 (Test Manager)**：负责测试进度的整体把控、资源的协调与分配、最终测试报告的出具及版本放行审批。
- **系统测试工程师 (System QA)**：负责搭建多种 Linux 操作系统环境、执行功能测试及异常容错测试。
- **配置与部署专家 (Deployment Expert)**：负责测试各类复杂的 YAML 配置文件组合、多节点拓扑下的高可用集群验证，以及离线 Registry 的场景模拟。
- **研发工程师 (Dev Support)**：实时响应测试过程中抛出的异常，协助排查深层架构（如 Pipeline、SSH Connector）相关的疑难问题。

## 3. 测试阶段与时间表 (Test Phases and Timeline)

整个测试周期设定为 **10个工作日**，划分为五个核心阶段：

### 3.1 阶段一：基础功能与冒烟测试 (Day 1 - Day 2)
- **目标**：验证代码可执行性、参数解析正确性以及基础设施。
- **测试内容**：
  - 验证 `version`, `help` 及所有的无效输入防御。
  - 验证 `create manifests` 针对各类 Kubernetes 版本（含高低版本）及运行时的输出合法性。
  - 核心 YAML 文件的 Schema 静态校验测试。
- **硬件要求**：个人工作站或单台云服务器 (2C4G, 联网)。
- **交付物**：冒烟测试通过报告。

### 3.2 阶段二：离线资源管理与分发测试 (Day 3 - Day 4)
- **目标**：覆盖最重要的特性之一——离线物料准备与 Registry 管理。
- **测试内容**：
  - 使用 `download` 验证不同 OS 架构下 BOM 清单拉取的精准度。
  - 执行 `create iso`，对 Rocky 9 及 Ubuntu 22 进行双架构 Docker 构建。
  - 配置 `spec.registry.enable`，测试 `create registry`, `push images` (并发推送及双镜像机制)，以及 `delete registry` 清理。
- **硬件要求**：需要一台大带宽并安装 Docker 的打包机，至少 100GB 硬盘空间。
- **交付物**：离线资源包(BOM)完整性验证报告。

### 3.3 阶段三：全生命周期部署核心测试 (Day 5 - Day 7)
- **目标**：验证在真实多节点环境中的创建、扩缩容与卸载闭环。
- **测试内容**：
  - **在线单节点**：一键式 `create cluster`。
  - **离线多节点**：利用阶段二准备的包，在完全无外网环境下部署 1 Master + 2 Worker 拓扑。
  - **动态伸缩**：分别对 worker 角色和 control-plane 角色执行 `scale-out` 和 `scale-in`，验证节点驱逐 (drain) 逻辑的优雅性。
  - **集群销毁**：验证 `delete cluster` 的完整清理及 `--backup` 选项的触发机制。
- **硬件要求**：4台位于同一 VPC 且相互打通无外网的虚拟机 (每台 4C8G)。
- **交付物**：集群核心生命周期测试矩阵报告。

### 3.4 阶段四：灾备与维护专项演练 (Day 8 - Day 9)
- **目标**：验证运维场景下的变更与数据恢复机制。
- **测试内容**：
  - K8s 跨小版本及 etcd 版本平滑升级 (`upgrade`)。
  - 手动及过期的 CA 证书与节点证书续期 (`renew`)。
  - 在集群制造数据损坏后，通过快照进行完整还原 (`restore cluster`)。
  - 重载部分组件配置 (`reconfigure`) 以及执行完整的系统诊断 (`health`)。
- **硬件要求**：阶段三创建好的稳定多节点集群。
- **交付物**：灾难恢复及运维升级演练报告。

### 3.5 阶段五：架构与极端异常测试 (Day 10)
- **目标**：进行压力与容错验证，检测系统设计底线。
- **测试内容**：
  - 在部署关键节点（如 `kubeadm init` 阶段）模拟网络切断，检查重试与超时退出机制。
  - 篡改部分已下载的离线包，测试完整性校验失败后的终止提示。
  - 验证内部代码模块层级 `Pipeline` 调用 `Task`，`Task` 调用 `Step` 所产生的日志格式合规性与执行隔离性。
- **交付物**：最终测试总结报告与缺陷列表。

## 4. 交付物与评审 (Deliverables and Sign-off)
测试执行完毕后，测试团队需提交以下材料进行版本放行评审：
1. **测试用例执行状态表**：记录所有用例的 Pass/Fail 状态。
2. **缺陷记录表**：记录发现的所有 Bug 及修复回归状态。
3. **测试总结报告**：总结系统的总体质量情况，确认是否存在带伤上线问题。
