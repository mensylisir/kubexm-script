# KubeXM 测试计划

## 1. 测试计划概述
本计划将测试任务拆分为五个阶段，从基础功能、离线资源管理、核心生命周期到升级维护，最后进行架构层面的专项测试，确保逐步排查潜在问题并覆盖全部测试范围。

## 2. 第一阶段：环境初始化与基础命令测试
- **测试重点**：验证安装流程、环境变量配置和基础只读/输出命令。
- **目标内容**：
  - 测试 `kubexm version`, `kubexm help` 是否返回正确输出。
  - 测试 `kubexm create manifests` 命令在不同版本与类型（kubeadm/kubexm）参数下的依赖清单生成功能。
- **环境要求**：任意单机环境即可。
- **预计周期**：1天。

## 3. 第二阶段：离线资源管理与 BOM 测试
- **测试重点**：保障 KubeXM 的离线生态功能可用，验证下载与打包逻辑。
- **目标内容**：
  - **Download 测试**：验证 `download` 命令根据配置文件（不同 CNI，不同运行时）是否能够正确下载 `os`, `binary`, `image`, `helm` 资源并生成对应的 BOM 清单。
  - **ISO 构建测试**：执行 `create iso` 命令，涵盖多系统（如 `rocky9`, `ubuntu22`）和多架构（`amd64`, `arm64`），验证最终 ISO 的生成及内容。
- **环境要求**：一台拥有良好公网连接并已安装 Docker 的构建机。
- **预计周期**：2天。

## 4. 第三阶段：集群核心生命周期测试
- **测试重点**：验证在线部署、离线部署以及节点操作的完整工作流。
- **目标内容**：
  - **在线单节点**：使用 `create cluster` 自动完成下载和单节点集群初始化。
  - **离线多节点**：配合已下载的 `packages` 目录，执行 `create registry`, `push images` (支持 `--parallel` 并发与 `--dual` 策略)，随后离线执行 `create cluster` 部署 1 Master + 2 Worker 架构。
  - **扩缩容测试**：执行 `scale cluster`，对特定的 worker 或 control-plane 节点进行扩容 (`scale-out`) 和缩容 (`scale-in`)，验证 drain 的安全机制。
  - **卸载测试**：执行 `delete cluster` 和 `delete registry`，同时验证带有 `--backup` 参数时是否会在卸载前生成数据快照。
- **环境要求**：在线单机环境、多节点内网离线环境。
- **预计周期**：3天。

## 5. 第四阶段：集群维护与升级操作测试
- **测试重点**：验证在已有集群上进行的修改与灾备功能的有效性。
- **目标内容**：
  - **版本升级**：分别执行 `upgrade cluster`（K8s版本）和 `upgrade etcd`，验证各组件无停机滚动升级机制。
  - **证书管理**：执行 `renew kubernetes-ca`, `renew etcd-ca`, `renew kubernetes-certs`, `renew etcd-certs` 验证证书续期逻辑，并确认配置能够自动刷新。
  - **备份恢复**：通过 `backup cluster` 生成 etcd 快照，随后修改或破坏集群部分数据，使用 `restore cluster` 验证是否完全恢复。
  - **健康与重载**：使用 `health cluster` 检查各组件与节点状态；修改配置文件后执行 `reconfigure cluster`（指定不同 `--target`）使新配置生效。
- **环境要求**：已正常运行的多节点集群环境。
- **预计周期**：2天。

## 6. 第五阶段：架构规范与异常场景测试
- **测试重点**：检测 KubeXM 的容错能力及架构设计约束（Pipeline/Task/Step 及 Config 校验）的严密性。
- **目标内容**：
  - **配置校验失败**：故意在 `config.yaml` 引入 schema 错误，验证工具在最外层解析时直接退出，不产生脏数据。
  - **环境阻断与超时**：模拟 SSH 密码错误、连接超时等异常情况，测试 Connector 层的重试机制与退出行为。
  - **执行流程中断**：在中途任务（如某台节点因磁盘满导致失败）出错时，检查日志能否清晰定位到具体的 Task 和 Step，流程能否安全中止。
- **预计周期**：2天。
