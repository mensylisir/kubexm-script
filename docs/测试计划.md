# KubeXM 全参数覆盖深度测试计划 (Exhaustive Test Plan)

## 1. 计划宗旨与范围界定

为应对 KubeXM 对全栈 K8s 部署生态的极高兼容性需求，本测试计划突破常规冒烟测试范畴，建立起基于环境矩阵的大规模排列组合测试排期。它涵盖了所有的命令组合、各类参数变量（包括且不限于 `OS`, `K8s 版本`, `Container Runtime`, `CNI`, `LB`）、物理和网络架构（在线与完全离线），并验证从命令行解析至底层执行引擎（SSH Connector 及 Config Schema）的层层防御规范。

## 2. 软硬件环境准备及矩阵配置 (Matrix Setup)

为覆盖多种系统组合，测试基础设施要求如下资源的准备与连通：

| 测试拓扑编号 | 网络状态 | 操作系统 | 容器运行时 | K8s 负载均衡 | 节点结构 (角色分配) | 主要验证目标 |
| --- | --- | --- | --- | --- | --- | --- |
| **ENV-1** | 纯公网在线 | Ubuntu 22.04 (DEB) | containerd | Kube-VIP | 1 单机版混合节点 | 基础部署、命令联通、轻量级功能覆盖。 |
| **ENV-2** | 纯内网隔离 (Air-Gap) | Rocky Linux 9 (RPM) | Docker | HAProxy | 1 Master + 2 Worker + 1 Registry | 离线 BOM 分发能力、内网注册表推送 (`dual` + 并发)、多节点缩扩容。 |
| **ENV-3** | 弱网波动环境 | CentOS 7.9 (RPM) | crio / podman | Nginx | 3 Master + 3 Worker (高可用) | 架构鲁棒性、Connector 超时与重试机制、系统极限压力测试。 |

*注：以上环境的 K8s 版本将从 v1.25.x 至 v1.34.x 交叉选择，CNI 将交替采用 Calico、Cilium 及 Flannel 以组合验证。*

## 3. 测试阶段划分与日程执行表 (Phase & Schedule)

计划整体划分为 6 个阶段，总计耗时约 3 星期。每一个阶段对应 KubeXM 生命周期的特定切面，且包含架构要求的防御性拦截测试。

### 阶段一：配置防守及命令行边缘解析 (Days 1 - 2)
**负责人**：配置管理测试专家
- **测试范畴**：YAML 结构解析模块及所有带有特殊可选参数命令的执行边缘。
- **任务大纲**：
  - 构造包含非法数据类型（如给整型字段赋字符串）、包含不存在组件版本号的恶劣 `config.yaml`。执行 `kubexm create cluster` 等必须读取配置的命令，强制验证 Config Parser 能够在 Pipeline 发起前100%报错退出，绝对禁止触发底层操作。
  - 执行 `kubexm create manifests` 使用全矩阵变量（`--kubernetes-version`, `--kubernetes-type=kubeadm/kubexm`, `--cni`等），对生成的依赖文字清单做静态合规检查。

### 阶段二：全量资产下载及 BOM 精准度验证 (Days 3 - 5)
**负责人**：物料与包管理验证员
- **测试范畴**：包下载能力、ISO 打包以及清单文件 (`*.bom.txt`) 生成。
- **任务大纲**：
  - 在全网连通的环境，基于 ENV-2 与 ENV-3 的配置执行 `kubexm download --cluster=...`。验证 `--kubernetes-version`、`--container-runtime` 等显式命令行参数成功覆盖 YAML 配置，并提取正确的 `os-bom`, `binary-bom`, `image-bom`, `helm-bom`。
  - 执行 `kubexm create iso` 组合验证 `--with-build-os` (如 rocky9, ubuntu22) 及 `--with-build-arch` (amd64, arm64, 或双架构) 的打包能力，测试 `install.sh` 的合法性。

### 阶段三：离线注册表与高并发镜像流转 (Days 6 - 7)
**负责人**：系统 QA 工程师
- **测试范畴**：在物理隔离环境内启动本地 Registry 及利用内置推图工具分发镜像的能力。
- **任务大纲**：
  - 在 ENV-2 结构中，测试 `kubexm create registry` 和 `kubexm delete registry --force --delete-images` 的无缝部署与强制清理机制。
  - 重点测试 `kubexm push images`：分别携带 `--packages` 目录获取流、`--parallel=10` 极限并发及 `--dual` 重打标推送，进入 Registry 容器内核查 Catalog 是否包含两套完整架构的镜像文件。

### 阶段四：在线/离线集群生命周期构建及水平扩容 (Days 8 - 12)
**负责人**：集群自动化部署专家
- **测试范畴**：Kubernetes 集群 `create` 及节点池的动态管理 `scale`，并强制测试底层异常容错设计。
- **任务大纲**：
  - **Create 验证**：在 ENV-1 进行一键化全自动在线部署；在 ENV-2 执行严苛的物理隔离环境建站，要求部署期间零字节互联网流量出入。
  - **Scale Out/In 验证**：执行 `kubexm scale cluster --action=scale-out/scale-in --role=worker --nodes=...`；缩容前必须触发系统 K8s Drain API 保证业务 Pod 优雅驱逐，随后节点组件自动安全卸载。
  - **Connector 韧性压力测试**：在拉起任意阶段（例如初始化阶段）人为修改 `iptables` 切断 SSH 长达 15 秒，验证底层 Connector 的超时及轮询重试逻辑（而非卡死），重试用尽后必须清晰打印具体 `[Task]` 和 `[Step]` 错误。

### 阶段五：集群高级运维操作及灾备恢复 (Days 13 - 16)
**负责人**：高级 K8s SRE 及运维测试员
- **测试范畴**：涵盖 `upgrade`, `renew`, `backup/restore`, `reconfigure`, `health` 等长期运维刚需特性。
- **任务大纲**：
  - **Upgrade/Renew**：对 ENV-3 集群执行 `upgrade cluster` 及 `upgrade etcd` 滚动验证无损平滑替换能力。验证 `renew` 家族命令（`kubernetes-ca`, `etcd-ca`, `kubernetes-certs`, `etcd-certs`）能否成功轮换过期证书并动态让 API Server 和 Kubelet 生效。
  - **Disaster Recovery (DR)**：手动执行 `backup cluster --path=...` 创建 etcd 快照；人工进入节点物理删除核心数据，随即执行 `restore cluster` 校验集群完全恢复。并在执行 `delete cluster --backup=...` 时验证强制卸载前必须生成的连带备份快照。
  - **Health/Reconfigure**：修改容器运行时参数，利用 `reconfigure --target=runtime` 检查非侵入式的单组件配置重启；执行 `health --check=all` 查看状态报表正确反馈。

### 阶段六：测试总结、缺陷收敛及发布评审 (Days 17 - 18)
- 汇总所有全参数矩阵及边缘测试下产生的缺陷。缺陷实行分级处理，P0 (数据销毁、致命崩溃) 及 P1 (Schema 拦截失败、离线特性残缺) 级 Bug 解决率为 100% 作为此方案的唯一放行准出标准。
