# KubeXM 全参数覆盖深度测试计划 (Exhaustive Test Plan)

## 1. 计划目标

基于《KubeXM 全面深度的测试方案》，本测试计划明确定义了为期 **3周** 的执行排期与环境拓扑资源，确保代码库中多达 25+ 个的执行参数（`--action`, `--backup`, `--check` 等等）、各个架构分支流转以及环境隔离要求（纯公网/纯内网）得到有序的矩阵组合测试。

## 2. 硬件及拓扑环境配置 (Test Environments)

建立三大类基础环境池：
| 编号 | 环境特征 | 操作系统 | K8s架构及角色 | 主要承担测试阶段 |
| --- | --- | --- | --- | --- |
| **ENV-A** | 在线公网 | Ubuntu 22.04 | 1 混合单节点 | 基础指令解析、Schema错误输入拦截、在线单机生成、`manifests` 清单输出。 |
| **ENV-B** | 物理 Air-Gap (无外网) | Rocky Linux 9 | 1 Master + 2 Worker + 1 Registry | 依托 BOM 与离线 `packages` 的全离线拉起、Registry 的并发/双重推送（`--parallel`/`--dual`）、扩缩容节点驱逐。 |
| **ENV-C** | 弱网可控网络 | CentOS 7.9 | 3 Master + 3 Worker | Connector 断网超时重试验证、版本滚动升级 (`upgrade`)、全维度健康检查 (`health --check=all`) 及灾难恢复 (`backup`/`restore`)。 |

## 3. 分阶段执行排期 (Execution Schedule)

### Phase 1: 架构级 Schema 与底层命令行拦截 (Days 1 - 2)
- **重点关注代码模块**：`internal/config/validator/schema.sh` 及 `bin/kubexm`
- **执行任务**：
  - 发动全量命令传入错误的标志位参数（如拼写错误），确保 CLI 入口正常抛出异常及帮助信息。
  - **Schema 零容忍测试**：在 YAML 中填入不合法格式（比如数字替代字符串），使用 `create cluster` 触发加载并验证 Pipeline 中止于预检层，阻止产生任何后续的 Connector 调用和实质性网络通信。
  - 使用全矩阵（覆盖 `--kubernetes-version`, `--kubernetes-type`, `--cni`, `--arch` 等）执行 `create manifests` 打印核对表。

### Phase 2: 全资产离线准备及 ISO 构建验证 (Days 3 - 5)
- **重点关注代码模块**：`internal/pipeline/assets/download.sh`, `iso.sh`
- **执行任务**：
  - 测试 `download`：使用命令行参数 `--kubernetes-version=v1.27.2 --container-runtime=docker` 强制覆盖 `config.yaml` 内的默认值，核实下载产物与生成的所有 `.bom.txt` 清单文件的完全一致性。
  - 测试 `create iso`：使用穷举组合参数 `--with-build-os=rocky9/ubuntu22`, `--with-build-arch=amd64/arm64`, `--with-build-all`，验证利用宿主机 Docker 生成各类系统离线安装介质的可行性和内部的 `install.sh`。

### Phase 3: 本地化 Registry 高并发流转 (Days 6 - 7)
- **重点关注代码模块**：`internal/pipeline/cluster/registry.sh`, `internal/pipeline/assets/push_images.sh`
- **执行任务**：
  - 在 ENV-B 中通过解析角色表启动 Registry（执行 `create registry`）。
  - 使用严苛条件执行 `push images --cluster=mycluster --packages --parallel=15 --dual`，测试高并发限流下，基于离线包的镜像推图，以及 `--dual` 功能对内外网 Tag 的重打标动作。
  - 测试 `delete registry --force --delete-images` 验证资源彻底清理无遗漏。

### Phase 4: K8s 集群双分支部署及架构重试容错 (Days 8 - 12)
- **重点关注代码模块**：`internal/pipeline/cluster/create_cluster.sh`, `scale_*.sh`, `internal/connector/`
- **执行任务**：
  - **分支一：在线部署** (ENV-A)。执行 `create cluster`，验证流畅的在线流转。
  - **分支二：离线隔绝部署** (ENV-B)。在严格切断外网下，使用已拉取的 `packages` 及本地 Registry 跑通 3 节点集群拉起全过程。
  - **Connector 异常自愈验证**：在集群拉起的某个原子的 `Step` 执行时，手动利用防火墙掐断 SSH 通信。监控日志确认工具进行规定的超时倒数与轮询重试（而非崩溃或死锁），并在最终失败时吐出精确的 `[Task]/[Step]` 调用栈错误。
  - **扩缩容指令群验证**：执行 `scale cluster --action=scale-out/scale-in --role=worker --nodes=...`。验证 `--action=scale-in` 时前置必须引发 K8s 原生 API 的 `drain` 动作，驱逐 Pods 后再卸载节点环境。

### Phase 5: 高级无缝运维套件深度演练 (Days 13 - 16)
- **重点关注代码模块**：`upgrade_*.sh`, `renew_*.sh`, `backup.sh`, `restore.sh`, `reconfigure.sh`, `health.sh`
- **执行任务**：
  - 在 ENV-C 集群中验证参数兼容性拦截，之后执行 `upgrade cluster --to-version=v1.33.0` 及 `upgrade etcd --to-version=3.5.15`，观测集群滚动升级间的业务中断率。
  - 测试四个续期分支命令：`renew kubernetes-ca / etcd-ca / kubernetes-certs / etcd-certs`。
  - 利用 `reconfigure cluster --target=runtime/etcd/cni/lb/addons/hosts` 单独更新并重载某一局部组件的配置。
  - 执行 `health cluster --check=node/component/workload/all` 生成系统深度探针报表。
  - 强制灾难破坏节点存储，使用 `--path` 指向的备份数据文件执行 `restore cluster` 验证回滚闭环；在 `delete cluster` 时传入 `--backup` 验证其灾备阻断机制。

### Phase 6: 验收评估及回归 (Days 17 - 18)
- 执行测试全案代码的回归 (`make test`)。审查所有 P0/P1 （尤其是穿透 Schema 防御及离线下载失效）漏洞是否彻底闭环。
