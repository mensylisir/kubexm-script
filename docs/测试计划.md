# KubeXM 全参数与组件深潜测试计划 (Exhaustive Test Plan)

## 1. 计划目标
本测试计划设计为期 **3周** 的排期，打破仅验证表面逻辑的局限，深入探视 KubeXM 底层的 `internal/step/` 组件执行。计划不仅要求验证命令行中全量提取出的 **30+ 项 flag** (`--action`, `--dual`, `--with-build-os` 等)，还要模拟执行期间针对核心 CNI 网络模块及证书 (Certs) 模块的边缘阻断攻击。

## 2. 三维测试拓扑与组件集成验证矩阵 (Test Environments)

| 拓扑标识 | 网络层级 | K8s架构配置 | 验证重点组件域 (internal/step/) | 核心参数验证面 |
| --- | --- | --- | --- | --- |
| **ENV-A** | 纯公网高速 | containerd + Cilium | `download`, `iso`, `cni/cilium_delete` | `--with-build-os`, `--cni`, `--output-format` |
| **ENV-B** | 物理 Air-Gap (断网) | docker + Calico | `registry`, `images/packages_concurrent`, `kubeadm`, `cni/calico_delete` | `--parallel`, `--dual`, `--manifest`, `--target-registry`, `--packages-dir` |
| **ENV-C** | 异常干扰环境 | crio + Flannel | `certs/kubeadm`, `certs/etcd`, `etcd`, `cni/flannel_delete` | `--cacert`, `--cert`, `--key`, `--endpoints`, `--force`, `--backup`, `--path` |

## 3. 分阶段深度执行排期

### 阶段一：防御拦截与渲染全集压测 (Days 1 - 2)
**目标**：测透 `manifests` 清单渲染及 Schema 拦截机制。
- **任务大纲**：
  - 交替插入组合：`--kubernetes-version=v1.x`, `--kubernetes-type=kubexm/kubeadm`, `--cni`, `--arch=amd64/arm64`。输出 JSON (`--output-format=json`) 核查结构体。
  - 测试传入不支持的 OS 版本或不支持的 `--cni` 名称，观察防守逻辑的报错抛出。

### 阶段二：全资产隔离编译与精准下发 (Days 3 - 5)
**目标**：测试下载及 `create iso` 工具链。
- **任务大纲**：
  - 执行 `download` 追加 `--kubernetes-version`、`--container-runtime` 强制覆写 YAML，校验生成的 4 种 `*.bom.txt`。
  - 对 `create iso` 使用全部子参数：`--with-build-os`, `--with-build-os-version`, `--with-build-arch`, `--with-build-all`, `--with-build-local`，产生多维度的 ISO 构建并校验。

### 阶段三：Registry 并发流转与重打标 (Days 6 - 7)
**目标**：测试内网镜像分发的高并发限制及逻辑分支。
- **任务大纲**：
  - 测试 `push images --packages --packages-dir=xxx --parallel=20 --dual --list=xxx.txt`。
  - 附加 `--manifest` 参数生成跨架构清单索引，附带 `--target-registry` 指定异地靶机，压测本地调度引擎内存峰值。
  - 使用 `--force` 与 `--delete-images` 联动销毁，相当于直接按下无条件强制清理按钮。

### 阶段四：核心装配及节点动态驱逐 (Days 8 - 12)
**目标**：验证 `create cluster`, `delete cluster` 及 `scale cluster` 底层组件通信。
- **任务大纲**：
  - 对接外部 etcd 时，将 TLS 校验参数 `--endpoints`, `--cacert`, `--cert`, `--key` 投入配置，测试安全双向通讯阻断响应。
  - 利用 `--action=scale-in`, `--role=worker`, `--nodes` 模拟控制台“退线驱逐按钮”，严格验证 K8s API 执行业务层级的迁移。
  - 在销毁期间带上 `--backup`，测试在底层触发灾备逻辑快照落地。强制清理集群后，检查 `cni/calico_delete` 或 `flannel_delete` 脚本是否被准确调用以摧毁宿主网桥残留。

### 阶段五：组件热更及内部证书展期 (Days 13 - 16)
**目标**：测试 `internal/step/certs/*` 及其它运维逻辑组件套件的准确流转。
- **任务大纲**：
  - 使用 `upgrade cluster/etcd` 与 `--to-version` 验证安全滚动机制。
  - 强制对准 ENV-C 集群进行证书轮调 (`renew kubernetes-ca / etcd-ca / kubernetes-certs / etcd-certs`)。底层必须严格执行 `rotate_collect -> rotate_generate_new -> distribute_pki...` 等一系列组件步骤。校验对应 API 服务重启且未断连。
  - 执行 `reconfigure cluster --target=...` 及 `health cluster --check=...` 检测各级探针正确返回。
  - 通过指定 `--path` 参数重构恢复被破坏的数据源。

### 阶段六：容错性架构破坏测试与发布回归 (Days 17 - 18)
- 在上述步骤间隙中，任意切断网卡造成超时错误。审查引擎框架是否依然保持重试倒数、超时后抛出标准层级堆栈日志，绝不允许 bash 层卡死；待所有 P0/P1 防守（包括对 `--force` 参数拦截验证）通过后输出最终报告。
