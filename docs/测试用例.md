# KubeXM 极详尽全量测试用例设计 (Exhaustive Test Cases)

本文档包含了针对 KubeXM 17+ 个子命令及其几十项变种参数矩阵的执行验证。该用例设计在验证“幸福路径”(Happy Path) 的同时，极其重视针对其架构体系（Config Schema 约束，Pipeline/Task/Step 设计约束，SSH Connector 超时与重试设计）在恶劣网络与错误输入环境下的鲁棒性边界。

---

## 一、 系统架构边界与基础解析防守测试 (Architecture & Core Defense)

### TC-ARCH-01: Config Schema 非法类型穿透防守
- **测试目的**：验证底层解析器严格遵循静态架构规范，防止由于错误 YAML 字段在部署途中引发未知 Bash 报错。
- **前置条件**：在 `conf/clusters/mycluster/config.yaml` 中，将需要布尔类型的 `spec.registry.enable` 改写为整型 `1`；将 `kubernetes_version` 错填为不合法的 `12.3.4`。
- **执行命令**：`kubexm create cluster --cluster=mycluster`
- **预期结果**：工具应该在零时间差内在预处理的 Config Pipeline 中断，清晰抛出 "Schema Validation Failed: Invalid type for enable/Invalid version format"，必须绝对拦截，没有任何 SSH 连接产生。

### TC-ARCH-02: Connector 弱网抖动及超时自愈
- **测试目的**：验证底层 Runner / Connector 执行引擎符合架构容错规范，防止死锁卡死。
- **前置条件**：通过 `iptables -A INPUT -p tcp --dport 22 -j DROP` 在部署某关键中间步骤时（例如拉取镜像中），模拟目标机长达 20 秒的连接丢失。
- **执行步骤**：观察正在执行的 `kubexm scale cluster ...`。
- **预期结果**：工具探测到断联后不崩溃亦不无休止挂死，而是开始在终端打印 `[Connector] SSH Connection dropped, retrying in 5s...`，重试次数内恢复网络后，可继续未竟的 Pipeline 操作。若恢复失败则应抛出明确的 `[Step] xyz failed due to Max Timeout` 报错退出。

## 二、 `download` 及 `create` (非集群系) 资源管理命令组

### TC-CMD-DL-01: 参数驱动的全量及覆盖式离线下载 (`download`)
- **执行命令**：
  1. 默认解析：`kubexm download --cluster=mycluster`
  2. 极度覆盖：`kubexm download --cluster=mycluster --kubernetes-version=v1.27.2 --container-runtime=crio --cni=cilium`
- **预期结果**：
  - 测试1：基于配置文件的 BOM 列表将被准确下载至 `.kubexm/downloads/`。
  - 测试2：无视配置文件设置，下载包将由指定版本接管，下载目录中应生成 crio 相关的依赖而非 docker。

### TC-CMD-C-01: `create manifests` 全矩阵参数预览
- **执行命令**：`kubexm create manifests --kubernetes-version=v1.34.2 --kubernetes-type=kubexm --arch=arm64`
- **预期结果**：能够按要求的架构、类型计算出极度精确的组件清单，包含所有的镜像（带有精确 tag）、系统二进制文件地址，且不报任何语法错。

### TC-CMD-C-02: 多端环境构建 `create iso`
- **执行命令**：
  1. `kubexm create iso --with-build-os=rocky9 --with-build-arch=amd64`
  2. `kubexm create iso --with-build-os=ubuntu22 --with-build-all`
- **预期结果**：使用宿主 docker 正确挂载，分别产出对应 RPM 系及 DEB 系合法包含 `install.sh` 的多架构自举离线引导 ISO，供纯物理隔离网络使用。

## 三、 私有注册表与并发推送命令组 (`create registry`, `push images`, `delete registry`)

### TC-CMD-REG-01: 内网私有注册表建联与反清理
- **执行命令**：
  1. 安装：`kubexm create registry --cluster=mycluster`
  2. 卸载：`kubexm delete registry --cluster=mycluster --force --delete-images`
- **预期结果**：安装时从角色表精准解析到目标机并拉起 Registry（提供 HTTP 5000 服务）；卸载时跳过任何确认对话，停止该容器进程并连带删除宿主机残留的挂载映射数据。

### TC-CMD-PUSH-01: 极限并发与双 Tag 机制推送 (`push images`)
- **前置条件**：离线资源 `packages/images` 存有数百个下载好的基础组件镜像。
- **执行命令**：`kubexm push images --cluster=mycluster --packages --parallel=15 --dual`
- **预期结果**：工具正确启动内部并发调度（最高 15 线程限流）。`--dual` 功能被验证：Registry 不仅保有外网前缀如 `k8s.gcr.io/coredns:v1` 的原始镜像，还能看到被改写并重命名为内网路径的镜像记录。

## 四、 核心集群生命周期命令组 (`create cluster`, `delete cluster`)

### TC-CMD-LIFECYCLE-01: 在线公网一键生成与离线架构隔绝生成
- **执行命令**：`kubexm create cluster --cluster=mycluster`
- **预期结果**：
  - **分支一（在线）**：自动融合 `download` 的依赖解析逻辑并在线连接软件源流转，所有 Control Plane, CNI 插件被配置成功且 Node 为 Ready 态。
  - **分支二（离线 Air-Gap）**：前置断开宿主机网卡，确认纯依托阶段二生成的 local `packages` 及本地 Registry，最终完整拉起高可用架构（例如 3 Master + 3 Worker）。

### TC-CMD-LIFECYCLE-02: 强制卸载及备份拦截 (`delete cluster`)
- **执行命令**：
  1. 常规拒绝测试：执行 `kubexm delete cluster --cluster=mycluster` 并输入 `n`。预期中断进程保护数据。
  2. 灾备销毁：执行 `kubexm delete cluster --cluster=mycluster --backup=/opt/safe_etcd.db --force`。预期生成灾备文件在先，彻底卸载并清理 CNI, kubelet 配置残留于后。

## 五、 集群水平伸缩及节点漂移指令 (`scale cluster`)

### TC-CMD-SCALE-01: 角色动态定向缩扩容与优雅驱逐测试
- **执行命令**：
  1. 扩容：`kubexm scale cluster --cluster=mycluster --action=scale-out --role=worker --nodes=node-4,node-5`
  2. 缩容：`kubexm scale cluster --cluster=mycluster --action=scale-in --nodes=node-4`
- **预期结果**：扩容能够并行处理多台加入过程；**缩容操作的核心验证点**在于：工具必须前置调用 k8s 原生 API 执行 `drain` 操作（且正确规避 Daemonset 冲突），等待 Pods 优雅迁移完毕后方可执行目标主机的组件拆除与 k8s API remove-node。

## 六、 高级无缝运维套件 (Upgrade, Renew, Reconfigure, Health, Backup/Restore)

### TC-CMD-OPS-01: `upgrade` 滚动平滑替换测试
- **执行命令**：
  1. K8s 更新：`kubexm upgrade cluster --cluster=mycluster --to-version=v1.33.0`
  2. etcd 更新：`kubexm upgrade etcd --cluster=mycluster --to-version=3.5.15`
- **预期结果**：系统强制执行：版本间断校验（阻断跨越式升级）-> Control Plane 迭代 -> Worker 迭代 -> Addons 刷新。期间使用压力测试脚本监控外网路由不可断流。

### TC-CMD-OPS-02: `renew` 全维度细分证书延期
- **执行命令**：依次对集群实施 `kubexm renew kubernetes-ca/etcd-ca/kubernetes-certs/etcd-certs --cluster=mycluster` 四条子命令。
- **预期结果**：对叶子节点和根节点进行隔离验证，测试其修改时间戳刷新。确保证书重建完成后，相对应的组件 (apiserver 或 etcd) 被动态重启并加载新秘钥。

### TC-CMD-OPS-03: `reconfigure` 局部组件配置热重启
- **执行命令**：更改 YAML 的容器运行时注册表源，执行 `kubexm reconfigure cluster --cluster=mycluster --target=runtime`。
- **预期结果**：只有底层的 docker/containerd 服务重载配置被刷新并触发 daemon-reload 及重启，不发生任何集群级的干扰动作。针对 `target` 参数的其它变量 (etcd, cni, lb, addons, hosts) 作同级测试。

### TC-CMD-OPS-04: `health cluster` 深层探测诊断
- **执行命令**：分别执行 `kubexm health cluster --cluster=mycluster --check=node/component/workload`。
- **预期结果**：根据不同的 `--check` 深度返回不同纬度信息（如 Node 验证内核及负载状态；Component 验证服务存活性；Workload 验证核心 pod 的 restart 次数及可用态），表格打印结构无乱码。

### TC-CMD-OPS-05: 容灾恢复 `backup/restore cluster` 兜底演练
- **执行命令**：`kubexm restore cluster --cluster=mycluster --path=/opt/safe_etcd.db`
- **预期结果**：在 etcd 文件数据完全人为破坏后，仅凭此前生成的 `.db` 备份快照还原所有 K8s ConfigMap、Deployments 状态及集群元数据。
