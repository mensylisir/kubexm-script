# KubeXM 极详尽全量测试用例设计 (Exhaustive Test Cases)

本测试用例穷举了代码中支持的 **所有主命令、所有子命令、近 30 种参数（Flags）、及其交叉逻辑**，并严加设计了针对系统基础架构（Pipeline、Config Validator、Connector）边界突破的反制测试，将所有的操作映射为精准的命令行。

---

## 一、 系统架构边界与基础解析防守测试 (Architecture & Core Defense)

### TC-ARCH-SCHEMA-01: YAML Config Schema 零容忍拦截
- **测试目的**：验证预处理器对一切 YAML 不规范写入的精准驳回。
- **前置条件**：在 `config.yaml` 内，将布尔字段 `enable` 误写成 `"yes"`，将 LB 策略设为非法值 `load_balancer: nonexistent-type`。
- **执行命令**：`kubexm create cluster --cluster=mycluster`
- **预期结果**：相当于触发内置的安全熔断“按钮”，拦截流程。控制台必须立刻报错并中断，**不得建立任何外部 SSH 连接或改变目录**。

### TC-ARCH-CONN-01: Connector 弱网抗断自愈与层级溯源
- **测试目的**：验证 SSH 连接引擎超时自愈及标准化溯源日志抛出。
- **前置步骤**：在部署执行某一关键 Step 时，人为使用 `iptables -A INPUT -p tcp --dport 22 -j DROP` 阻断跳板机的 SSH 回报包长达 20 秒。
- **预期结果**：工具不会卡死。自动探测到 Drop 状态并触发重试倒数逻辑；若重试阈值用尽后，控制台抛出严格规范的堆栈日志，如 `[Pipeline] Cluster Install -> [Task] Network -> [Step] Download File -> Timed out`。

## 二、 资产管理与环境预建命令组 (`download`, `create manifests`, `create iso`)

### TC-ASSET-MANIFEST-01: 多维变量全景推演清单
- **执行参数组合**：`kubexm create manifests --kubernetes-version=v1.34.2 --kubernetes-type=kubeadm --arch=arm64 --cni=cilium --output-format=json`
- **预期结果**：根据传入的各类细分类型准确在控制台（以合法的 JSON 格式打印）演算出所需拉取的容器镜像 Tag 清单、系统底层二进制链接及 Helm 包列表。

### TC-ASSET-DOWNLOAD-01: 参数覆盖执行与离线物料准备
- **执行参数组合**：`kubexm download --cluster=mycluster --kubernetes-version=v1.27.2 --container-runtime=docker`
- **预期结果**：CLI 中的标志参数具有最高权级，直接覆写 YAML 设置。最终保存在 `.kubexm/downloads/` 内的 `os-bom.txt`, `binary-bom.txt`, `image-bom.txt` 清单内容与下载的物理包字节一致。

### TC-ASSET-ISO-01: 跨端 ISO 自举介质生成全参测试
- **执行参数组合**：
  1. `kubexm create iso --with-build-os=ubuntu22 --with-build-os-version=22.04 --with-build-arch=amd64`
  2. `kubexm create iso --with-build-all`
  3. `kubexm create iso --with-build-local`
- **预期结果**：无论是驱动本地源打包还是通过挂载 Docker 打包，均成功生产携带 `install.sh` 及完整包依赖的多架构/指定架构引导介质 `.iso`。

## 三、 私有注册表及高并发镜像路由推移 (`create/delete registry`, `push images`)

### TC-REG-LIFECYCLE-01: 注册表生灭与无交互清除
- **执行参数组合**：
  1. `kubexm create registry --cluster=mycluster`
  2. `kubexm delete registry --cluster=mycluster --force --delete-images`
- **预期结果**：按 `host.yaml` 精准下发；销毁时 `--force` 绕过确认拦截（等效跳过确认交互按钮），`--delete-images` 从宿主机物理层面抹除所有 Registry 挂载目录，不留残影。

### TC-REG-PUSH-01: 并发限流、双标打号与多架构关联
- **执行参数组合**：`kubexm push images --cluster=mycluster --packages --packages-dir=packages/images --parallel=20 --dual --list=/opt/custom.txt --manifest --target-registry=hub.local.net:5000`
- **预期结果**：
  - 工具使用指定的 `--list` 和 `--packages-dir` 提取镜像，并将并发控制在 20 线程以内。
  - 推送至目标 `--target-registry` 仓库，且因 `--dual` 功能产生保留原官方 Tag 和被重命名打标的两份镜像实体。
  - `--manifest` 触发合并创建多架构 Docker Manifest 索引指令。

## 四、 核心集群部署架构与证书网关接驳 (`create/delete cluster`)

### TC-CLUS-CREATE-01: 公网与 Air-Gap 双端环境极限界部署
- **公网分支**：默认执行 `kubexm create cluster --cluster=mycluster`。
- **断网隔离分支 (Air-Gap)**：在阻断外网（拔网线级别）的 3 节点内网运行。确保 Pipeline 不含向外部探针的 `curl` 请求，仅依赖内网 Registry 和预编译包将 K8s 所有核心层拉平到 `Ready`。

### TC-CLUS-TLS-01: 外部组件安全鉴权流对接
- **执行参数组合**：在配置文件或通过追加命令行参数时使用 `--endpoints=https://ext-etcd.local:2379 --cacert=/etc/pki/ca.crt --cert=/etc/pki/client.crt --key=/etc/pki/client.key`
- **预期结果**：在组件（如 API Server 连外部 etcd）通信初始化时，必须强制加载并启用双向 TLS 握手验证，防止不安全的 HTTP 明文传输。

### TC-CLUS-DELETE-01: 强行下线带数据灾备
- **执行参数组合**：`kubexm delete cluster --cluster=mycluster --force --backup=/opt/kubexm-safe.db`
- **预期结果**：一旦工具捕获到 `--backup` 命令，它将重组任务队列，必须先保证产生合法数据块的快照落地，然后再粗暴摧毁 K8s 服务进程与网络虚拟设备。

## 五、 集群自动调度与安全缩扩容 (`scale cluster`)

### TC-CLUS-SCALE-01: `--action` 判定与节点安全退线 `drain` (关键交互)
- **扩容指令**：`kubexm scale cluster --cluster=mycluster --action=scale-out --role=worker --nodes=worker-4`
- **缩容指令**：`kubexm scale cluster --cluster=mycluster --action=scale-in --role=worker --nodes=worker-4`
- **预期结果**：
  - 缩容等效于在管理界面按下“节点安全下线”。它绝对不允许直接删除文件。
  - 工具内必须通过 API 发起 `drain` 命令驱逐用户负载，监控 Pods 漂移状态，在所有业务被平稳移交后，才卸载并注销该节点。

## 六、 运维底盘无感更新及重载诊断体系 (Upgrade, Renew, Reconfigure, Health, Backup/Restore)

### TC-OPS-UPGRADE-01: 防跌断滚动更迭
- **执行参数组合**：分别运行 `kubexm upgrade cluster --cluster=mycluster --to-version=v1.33.0` 与 `kubexm upgrade etcd --cluster=mycluster --to-version=3.5.15`
- **预期结果**：严格对 `--to-version` 进行版本跨度检查，采用“逐个替换、等待恢复”的无中断策略滚动重启控制面及数据面，保持服务外连正常。

### TC-OPS-RENEW-01: 精准颗粒度证书吊销刷新
- **执行参数组合**：依次测试 `kubexm renew kubernetes-ca / etcd-ca / kubernetes-certs / etcd-certs --cluster=mycluster`
- **预期结果**：能针对性地延期基础 CA 及派生叶子节点证书。证书刷新落盘后，对应的（apiserver/kubelet 等）组件必定被自动重启重新载入内存。

### TC-OPS-RECONF-01: 局部靶向组件热更新
- **执行参数组合**：`kubexm reconfigure cluster --cluster=mycluster --target=runtime` (亦需轮替 etcd, cni, lb, addons, hosts)
- **预期结果**：单独修改部分属性，工具仅在后端对匹配的 `target` 执行重新下发配置并施加类似 `systemctl reload` 的轻柔手段，不干涉整体网格。

### TC-OPS-HEALTH-01: 全层级多态诊断报告
- **执行参数组合**：`kubexm health cluster --cluster=mycluster --check=node/component/workload/all --output-format=table`
- **预期结果**：工具对 OS 负载状态、kube-system 核心存活性及关键业务重启频率分别提取探针结果，最终排版成高度可读或便于机器解析（JSON）的报表。

### TC-OPS-RESTORE-01: 完全破坏后的单点回溯
- **执行参数组合**：`kubexm restore cluster --cluster=mycluster --path=/opt/safe_etcd.db`
- **预期结果**：手动删库后，将先前的冷备份 `.db` 指定后执行回滚，校验此前的全部业务状态（Deployments/Secrets 等）被一字不差地找回。
