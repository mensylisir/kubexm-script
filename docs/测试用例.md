# KubeXM 极限界防守与全指令参数执行测试用例 (Exhaustive Test Cases)

本文档严密涵盖了 `bin/kubexm` 中全部 17+ 种主干/子命令、30 多项控制参数标志（Flags）以及与之相互映射的内部组件模块（位于 `internal/step/`，如 `certs/kubeadm`, `cni/calico_delete.sh` 等）。测试用例设计了正常的流程、交互替代（如利用 `--force` 选项当做图形界面的“强制同意”按钮）以及异常阻断拦截。

---

## 模块一：架构防守与配置穿透免疫 (Architecture Defenses)

### TC-ARCH-01: Config Schema 非法数据与类型防守
- **测试目的**：验证底层解析器拒绝一切不合法指令的静态能力。
- **前置条件**：修改 YAML，设定不受支持的 `--cni` 名称、或在布尔值处输入文本。
- **执行命令**：`kubexm create cluster --cluster=mycluster`
- **预期行为**：系统预处理器直接报错并阻断，不得引发对目标的远程通讯；绝对不能有未知 Bash 错乱或 `command not found` 渗漏至终端。

### TC-ARCH-02: SSH Connector 重试与自愈框架
- **测试目的**：验证连接引擎 (Connector) 超时机制的韧性。
- **执行过程**：在执行安装原子命令期间，通过主机 `iptables` 丢弃跳板机到 Target 节点的网络回报包，导致阻塞达 20 秒以上。
- **预期行为**：终端挂起但未崩溃，引擎检测到阻塞后输出 `retrying in X seconds...`。超时耗尽后则逐级退出并标准打印 `[Pipeline] -> [Task] -> [Step] -> Timed out`，保障死锁自防机制有效。

## 模块二：离线资产打包与依赖计算 (Download & ISO)

### TC-ASSET-01: 参数覆盖模式的 BOM 演算
- **执行命令**：`kubexm download --cluster=mycluster --kubernetes-version=v1.27.2 --container-runtime=docker`
- **预期行为**：CLI 的参数覆盖等级最高，下载流水线必须放弃配置文件中的原有字段设置。核对生成的 `os-bom.txt`, `binary-bom.txt`, `image-bom.txt` 是否存在并精确匹配 1.27.2 及 docker 的组件结构。

### TC-ASSET-02: `--output-format` 输出兼容测试
- **执行命令**：`kubexm create manifests --kubernetes-version=v1.34.2 --kubernetes-type=kubeadm --arch=arm64 --cni=cilium --output-format=json`
- **预期行为**：根据传入的精准参数，不报错地演算出正确的容器镜像及文件 URL 地址，由于设置了 JSON 输出，必须保证 JSON 的强正确性供外部机器系统解析。

### TC-ASSET-03: ISO 构建全景参数组合
- **执行命令**：
  1. `kubexm create iso --with-build-os=ubuntu22 --with-build-os-version=22.04 --with-build-arch=amd64`
  2. `kubexm create iso --with-build-all`
  3. `kubexm create iso --with-build-local`
- **预期行为**：各指令组合下调用相应容器或在本地执行构建出含有底层源和自举引导 (`install.sh`) 脚本的 `.iso` 文件。

## 模块三：镜像管理并发流向 (Registry Lifecycle)

### TC-REG-01: 私有组件强制卸载 ("一键删除"交互等效)
- **执行命令**：
  1. `kubexm create registry --cluster=mycluster`
  2. `kubexm delete registry --cluster=mycluster --force --delete-images`
- **预期行为**：卸载时，`--force` 直接跳过交互式二次确认；配合 `--delete-images` 参数，直接联动清理位于宿主机深处的所有 registry 持久化挂载卷，不留残迹。

### TC-REG-02: 极限推图与双域打标
- **执行命令**：`kubexm push images --cluster=mycluster --packages --packages-dir=packages/images --parallel=20 --dual --list=/tmp/custom.txt --manifest --target-registry=hub.local`
- **预期行为**：
  - 测试内部的并发队列机制在 `--parallel=20` 线程下未导致资源内存溢出。
  - `--dual` 参数：推向库中不仅存有 `hub.local/docker.io/...` 等等外层 Tag 还有重定位的内网 Tag。
  - `--manifest` 成功触发合并不同系统架构的跨平台 Image Manifest 配置命令。

## 模块四：核心集群全向创建与接驳 (Cluster Create/Delete)

### TC-CLUS-01: Air-Gap 环境离网纯依赖拉起
- **执行过程**：在网卡被拔除的情况下，在 3 节点集群通过 `--cluster=mycluster` 触发拉起；配置完全指向内网搭建的包与注册表。
- **预期行为**：所有相关的底层 `internal/step/` 脚本不允许调用向公网进行寻址下载的代码。集群顺利拉平为 Ready 状态。

### TC-CLUS-02: 外部鉴权 API/TLS对接
- **执行命令**：带上参数 `--endpoints=https://ext-etcd.local:2379 --cacert=/ca.crt --cert=/client.crt --key=/client.key`
- **预期行为**：系统向独立存储（etcd）或外部 API 发起联络时强制进行高等级的双向 TLS 加密握手验证，防止不安全的连接配置建立。

### TC-CLUS-03: `cni/*_delete` 网络沙盒无痕拔除
- **执行命令**：`kubexm delete cluster --cluster=mycluster --force --backup=/opt/kubexm-disaster.db`
- **预期行为**：在探测到 `--backup` 后，将拆除次序调转，第一时间落地快照数据库。随后清理时底层脚本如 `cni/calico_delete.sh` 等必须被精确触发，清除主机网卡的残余 veth 虚拟网络设备。

## 模块五：动态安全伸缩与迁移 (Scale & Drain)

### TC-SCALE-01: 基于 `--action` 的驱逐按钮动作
- **扩容执行**：`kubexm scale cluster --cluster=mycluster --action=scale-out --role=worker --nodes=worker-4`
- **缩容执行**：`kubexm scale cluster --cluster=mycluster --action=scale-in --role=worker --nodes=worker-4`
- **预期行为**：扩容仅向新增机器下发环境；缩容则**严禁暴力直接剔除节点**，它必须发起原生的 K8s `drain` 操作（等价于管理员在界面点击的 "平滑安全下线" 按钮），待该节点所有存活 Pod 成功流转别处后，方执行物理卸载清理。

## 模块六：无感重载及底层证书网格 (Ops & Maintain)

### TC-OPS-01: 滚动跨度拦截
- **执行命令**：`kubexm upgrade cluster --cluster=mycluster --to-version=v1.33.0` 与 `upgrade etcd`
- **预期行为**：拦截不被支持的跨大版本号跳跃。随后逐步迭代 Master 节点及 Worker，保证在线应用仍能承接外部流量。

### TC-OPS-02: `certs/kubeadm` 内部证书生成及流转隔离
- **执行命令**：针对 `kubernetes-ca/etcd-ca/kubernetes-certs/etcd-certs` 依次交替执行 `kubexm renew ... --cluster=mycluster`。
- **预期行为**：验证深层逻辑，如 `certs/kubeadm/distribute_pki_k8s_copy.sh` 等脚本必须确保将刚由主控生成的新鲜证书严密下放至全量节点 `/etc/kubernetes/pki` 并完成恰当的挂载权限锁，随后相关服务须能无缝加载新证书。

### TC-OPS-03: 局部无侵入动态下发
- **执行命令**：`kubexm reconfigure cluster --cluster=mycluster --target=runtime` (同理可测 `etcd`, `cni`, `lb`, `addons`, `hosts`)
- **预期行为**：指定 `target` 后，工具只允许向相关特定的服务配置文件下发并触发类似 `systemctl reload` 的轻动作，全局的组件环境保持不中断。

### TC-OPS-04: 多形态矩阵体检系统
- **执行命令**：`kubexm health cluster --cluster=mycluster --check=node/component/workload/all`
- **预期行为**：依不同深度的探针命令，控制台应回显相应的（如负载过高告警、CrashLoopBackOff 指示等）详细报表。

### TC-OPS-05: 单点灾难召回点
- **执行命令**：`kubexm restore cluster --cluster=mycluster --path=/opt/safe_etcd.db`
- **预期行为**：数据存储灾难清空的情况下，仅依赖一个有效的 `--path` 备份文件即可强行召回历史架构环境。
