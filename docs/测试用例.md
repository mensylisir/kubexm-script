# KubeXM 全指令矩阵与架构底盘防守级测试用例 (Exhaustive Test Cases)

本文档严密地针对 `bin/kubexm` 中全部 17+ 种指令及 30 余项执行参数（Flags）作了用例编排。其中融入了对 `internal/step/` 核心包（如证书颁发、网络卸载脚本）底层执行机理的探索，同时明确标识了“等效交互按钮”（例如通过 CLI 的 `--force` 和 `--action` 实现图形界面的“确认通过”与“安全下线”按钮效果）的用例。

---

## 一、 系统架构边界与基础解析拦截 (Architecture & Defenses)

### TC-ARCH-SCHEMA-01: YAML 非法配置绝对阻击
- **测试目的**：验证工具是否具有防渗透、防误写的零容忍边界。
- **操作步骤**：手动污染配置文件（例如 `enable: "true"` 替代布尔值、填写不存在的负载均衡器名称）。
- **执行命令**：`kubexm create cluster --cluster=mycluster`。
- **预期结果**：相当于在发起申请的源头即被熔断，程序即刻退出并打印 Schema Validated Error。决不引发向服务器的底层连接和执行尝试。

### TC-ARCH-CONN-01: 跨节点 Connector 重试自愈模型
- **测试目的**：针对内部组件在调用 SSH Connector 时可能遭遇的弱网。
- **操作步骤**：在拉起关键组件（如执行到 `join_worker` 系列脚本）时，设置网络防火墙将目标节点 22 端口 DROP 控制在 20 秒。
- **预期结果**：工具挂起但绝不闪退崩溃，开始打印规范化倒数 `retrying in X seconds`。一旦倒数耗尽无响应，必须按照 `[Pipeline] -> [Task] -> [Step] -> Timed out` 层级逐一溯源报错抛出。

## 二、 资产管理与依赖准备工作台 (Asset & Build Workflow)

### TC-ASSET-MANIFEST-01: 参数覆盖生成定制化清单
- **执行参数**：`kubexm create manifests --kubernetes-version=v1.34.2 --kubernetes-type=kubeadm --arch=arm64 --cni=cilium --output-format=json`
- **预期结果**：验证在控制台以合法 JSON 打印清单。内部拉取的 Image Tag 及对应的 RPM/Debian 包必须基于 `v1.34.2`, `arm64` 且为 `cilium` 相关体系计算。

### TC-ASSET-DOWNLOAD-01: 参数越权与文件指纹验证
- **执行参数**：`kubexm download --cluster=mycluster --kubernetes-version=v1.27.2 --container-runtime=docker`
- **预期结果**：命令行传参必须具有覆盖 `yaml` 最高权级。下载至 `.kubexm/downloads/` 的缓存及其同级生成的四大 `.bom.txt` 清单文件，其字节和版本描述完全对标 v1.27.2 + docker。

### TC-ASSET-ISO-01: 容器级交叉编译打包测试
- **执行参数**：分别执行：
  1. `kubexm create iso --with-build-os=ubuntu22 --with-build-os-version=22.04 --with-build-arch=amd64`
  2. `kubexm create iso --with-build-all`
  3. `kubexm create iso --with-build-local`
- **预期结果**：调用容器技术成功为目标发行版生成含 `install.sh` 脚本和依赖库源的 Bootable ISO 镜像。

## 三、 私有仓库搭建与大并发吞吐 (Registry Lifecycle)

### TC-REG-01: 一键清盘与“强行按钮”模拟
- **执行参数**：
  1. `kubexm create registry --cluster=mycluster`
  2. `kubexm delete registry --cluster=mycluster --force --delete-images`
- **预期结果**：通过 `--force` 参数模拟 GUI 界面中的“直接确认无需提醒”操作，瞬间跳过一切交互询问；并由 `--delete-images` 参数联动，深度抹除宿主机相关的持久化镜像数据卷。

### TC-REG-02: 流控吞吐、重打标与跨构架合并推送
- **执行参数**：`kubexm push images --cluster=mycluster --packages --packages-dir=packages/images --parallel=20 --dual --list=/tmp/custom.txt --manifest --target-registry=hub.local`
- **预期结果**：
  - 测试 `--parallel=20` 的多线程节流防爆。
  - `--dual` 作为内网打号“按钮”，使 Registry 里既有原本的外网命名空间也有转换为目标仓库的内网前缀。
  - 利用 `--manifest` 将相关镜像推向指定的 `--target-registry` 并联合打包生成 OCI 多架构 Index。

## 四、 集群底层拉起与安全网关对接 (Cluster Control)

### TC-CLUS-CREATE-01: 完全断网 Air-Gap 物理隔离生境验证
- **操作步骤**：掐断测试集群对外的核心网卡及 DNS 解析路由，依托在同网段机器拉起的私有源进行部署。
- **执行命令**：`kubexm create cluster --cluster=mycluster`。
- **预期结果**：整个系统部署期间（底层涉如 `certs/kubeadm`、`kubelet`、各类网络部署环节）完全不受卡顿与寻址超时干扰，全部依赖解析至内网配置，成功全亮 `Ready`。

### TC-CLUS-CREATE-02: 边界 TLS 双向强鉴权握手
- **执行参数**：在安装带外接存储选项时，通过命令附带 `--endpoints=https://ext-etcd.local:2379 --cacert=/etc/ca.crt --cert=/etc/client.crt --key=/etc/client.key`
- **预期结果**：连接外部网络节点时强制进入双向 HTTPS TLS 通信校验，验证凭据错位时立刻中断对接进程以保护通讯。

### TC-CLUS-DELETE-01: 数据抢救性连带强制销毁
- **执行参数**：`kubexm delete cluster --cluster=mycluster --force --backup=/opt/safe_data.db`
- **预期结果**：检测到 `--backup` 标志存在，Pipeline 打破顺序，第一步立刻执行 etcd 快照保存至路径，确保灾备落盘后再使用底层如 `cni/calico_delete.sh` 等专项脚本实施集群网卡及组件的粗暴销毁。

## 五、 弹性智能编排与漂移安全 (Scale & Safe Drain)

### TC-SCALE-01: “安全下线按钮”逻辑映射
- **缩容执行**：`kubexm scale cluster --cluster=mycluster --action=scale-in --role=worker --nodes=worker-4`
- **预期结果**：参数 `--action=scale-in` 直接映射为高级别集群保护动作。执行前它必须发起原生 K8s `drain` 指令等待并核实所有的生产 Pod 被安全移走，若遭遇 PDB 卡死则须超时提醒，驱逐完毕才可发起机器卸载与回收。

## 六、 运维底层修复与深层管控 (Ops Suite)

### TC-OPS-UPGRADE-01: 防跌断与版本兼容滚动升级
- **执行参数**：针对组件交错运行 `kubexm upgrade cluster --cluster=mycluster --to-version=v1.33.0` 与 `upgrade etcd`
- **预期结果**：触发跨度核检拦截；按先 Master 后 Worker 顺序开展替换，严禁大面积中断服务流。

### TC-OPS-RENEW-01: CA 根及末端证书深层下发及展期
- **执行参数**：拆分测试 `kubexm renew kubernetes-ca/etcd-ca/kubernetes-certs/etcd-certs --cluster=mycluster`。
- **预期结果**：系统深层次调度（如调用 `internal/step/certs/kubeadm/distribute_pki_k8s_copy.sh`），把主控节点新颁发的证书严密且携带正确属主权限地下发至每个 Node。完成后相应的组件即刻刷新加载新证书。

### TC-OPS-RECONF-01: 定向组件热装载 (Hot Reload)
- **执行参数**：`kubexm reconfigure cluster --cluster=mycluster --target=runtime` （及轮番验证 `etcd,cni,lb,addons,hosts`）
- **预期结果**：定向修改，不全局中断，仅通知并执行（如相当于 `systemctl reload docker`）动作下发新配置文件。

### TC-OPS-HEALTH-01: 系统级全面健康报表
- **执行参数**：`kubexm health cluster --cluster=mycluster --check=node/component/workload/all --output-format=table`
- **预期结果**：按照指定的深浅探针，输出格式平稳且极易人读的对应表格诊断状况。

### TC-OPS-RESTORE-01: 单点溯回与数据召回
- **执行参数**：`kubexm restore cluster --cluster=mycluster --path=/opt/safe_data.db`
- **预期结果**：对受损或空转的新集群传入合法的 `.db` 包，验证工具能够覆盖性拉起此前所有 Deployments, Services 和集群历史环境。
