# KubeXM 测试用例详细设计

本文档详细列出了针对 KubeXM 的各个功能模块与核心命令的测试用例，覆盖在线/离线、单节点/多节点等分支处理逻辑以及对系统架构容错性的验证。

---

## 1. 基础与辅助命令 (Base Commands)

### TC-BASE-01: 查看版本信息
- **测试目的**：验证基本运行环境是否正确配置并输出版本号。
- **执行步骤**：运行 `kubexm version` 或 `kubexm -v`。
- **预期结果**：输出如 `kubexm version 0.1.0` 等版本信息，无错误返回。

### TC-BASE-02: 查看帮助信息
- **测试目的**：验证所有帮助信息及参数说明。
- **执行步骤**：运行 `kubexm help` 或直接运行 `kubexm`。
- **预期结果**：屏幕打印核心命令列表、子命令帮助信息及使用示例。

### TC-BASE-03: 生成依赖清单
- **测试目的**：验证系统能否通过配置正确解析各组件版本并生成资源列表。
- **执行步骤**：
  1. 运行 `kubexm create manifests --kubernetes-version=v1.34.2 --kubernetes-type=kubexm`。
  2. 运行 `kubexm create manifests --cluster=mycluster` 读取配置。
- **预期结果**：成功打印出相应版本的镜像、二进制和 Helm 包依赖清单，不发生崩溃。

---

## 2. 资源下载与打包 (Download / ISO)

### TC-DL-01: 默认配置离线资源下载
- **测试目的**：测试依据默认 `config.yaml` 自动下载完整依赖的能力。
- **前置条件**：配置目录 `conf/clusters/mycluster/config.yaml` 存在且网络畅通。
- **执行步骤**：运行 `kubexm download --cluster=mycluster`。
- **预期结果**：
  1. 资源被正确下载至 `.kubexm/downloads/` 或指定目录。
  2. 目录下成功生成完整的 `os-bom.txt`, `binary-bom.txt`, `image-bom.txt` 和 `manifest.txt` 文件。

### TC-DL-02: 参数覆盖式资源下载
- **测试目的**：验证命令行参数对配置文件的覆盖行为。
- **执行步骤**：运行 `kubexm download --cluster=mycluster --kubernetes-version=v1.27.2 --container-runtime=docker --cni=cilium`。
- **预期结果**：下载的资源版本和类型（K8s=v1.27.2, 容器为docker, CNI为cilium）均符合命令行参数，而非 yaml 文件中的默认值。

### TC-DL-03: ISO系统包构建
- **测试目的**：测试 OS 依赖离线源的打包功能。
- **执行步骤**：运行 `kubexm create iso --with-build-os=rocky9 --with-build-arch=amd64`。
- **预期结果**：利用 Docker 构建生成符合 Rocky Linux 9 环境的 ISO 文件，并在生成的挂载镜像内发现合法的源文件及安装脚本。

## 3. 本地注册表与镜像管理 (Registry / Push)

### TC-REG-01: 离线创建并部署 Registry
- **测试目的**：验证 Registry 自动安装和初始化流程。
- **前置条件**：集群配置文件内 `spec.registry.enable` 设置为 `true` 且定义了 registry 角色。
- **执行步骤**：运行 `kubexm create registry --cluster=mycluster`。
- **预期结果**：在目标节点上拉起 Registry 服务，且通过 API 访问健康端点返回 200 OK。

### TC-REG-02: 并发与双向推送镜像
- **测试目的**：测试镜像批量推送的稳定性和并发处理逻辑。
- **执行步骤**：运行 `kubexm push images --cluster=mycluster --packages --parallel=8 --dual`。
- **预期结果**：
  1. 工具读取 `packages/images/` 下的列表并使用 8 个并发线程推送。
  2. 目标 Registry 中出现原始和重命名后的双镜像文件。

### TC-REG-03: 删除注册表 (Force 模式)
- **测试目的**：验证 Registry 及其数据的清理。
- **执行步骤**：运行 `kubexm delete registry --cluster=mycluster --force --delete-images`。
- **预期结果**：无交互提示下，服务被停止，对应的容器和数据挂载目录被完全清理。

## 4. 核心生命周期管理 (Create / Scale / Delete)

### TC-LC-01: 在线创建单节点集群
- **测试目的**：验证联网状态下的全流程。
- **执行步骤**：在联网环境运行 `kubexm create cluster --cluster=mycluster`。
- **预期结果**：系统自动串联预检查、下载依赖、系统环境初始化、K8s 节点配置的 Pipeline，最后输出成功提示，执行 `kubectl get nodes` 时状态为 Ready。

### TC-LC-02: 离线多节点集群部署
- **测试目的**：验证利用离线包及 Registry 配置部署复杂集群的能力。
- **前置条件**：断开外部网络，只使用下载好的 `packages` 及本地搭建的 registry。
- **执行步骤**：运行 `kubexm create cluster --cluster=mycluster`，配置文件包含 1 个 Master 节点及 2 个 Worker 节点。
- **预期结果**：3台服务器全部部署成功，CNI（如 Calico）启动正常，Pod 和 Node 健康状态为 100%。

### TC-LC-03: 节点平滑扩容 (Scale-out)
- **测试目的**：测试只向特定角色添加节点的行为。
- **执行步骤**：运行 `kubexm scale cluster --cluster=mycluster --action=scale-out --role=worker --nodes=new-worker-1`。
- **预期结果**：主集群正常工作，新的工作节点执行完环境准备和加入动作后成功展示在集群节点列表中。

### TC-LC-04: 安全缩容与驱逐 (Scale-in)
- **测试目的**：验证集群剔除节点的优雅流程（Drain -> 移除节点 -> 清理配置）。
- **执行步骤**：运行 `kubexm scale cluster --cluster=mycluster --action=scale-in --nodes=new-worker-1`。
- **预期结果**：系统首先完成节点上 Pod 的驱逐（drain），然后从 K8s 列表中删除该节点，并在宿主机上执行系统组件卸载和清理。

### TC-LC-05: 带有备份的集群强制删除
- **测试目的**：验证删除操作及灾难恢复前置条件（etcd快照备份）。
- **执行步骤**：运行 `kubexm delete cluster --cluster=mycluster --backup=/opt/backup/etcd-snap.db --force`。
- **预期结果**：确认 `/opt/backup/etcd-snap.db` 已成功生成，随后所有节点组件被卸载，无残留进程。

## 5. 维护管理与系统修复 (Upgrade / Renew / Health / Reconfigure)

### TC-MAINT-01: Kubernetes 版本滚动升级
- **测试目的**：验证集群组件原地升级方案。
- **执行步骤**：运行 `kubexm upgrade cluster --cluster=mycluster --to-version=v1.33.0`。
- **预期结果**：执行流程严格依照：预检查 -> ControlPlane 升级 -> Worker 升级 -> Addons 升级，完成后检查 `kubectl version` 结果匹配。

### TC-MAINT-02: ETCD 节点升级
- **测试目的**：针对核心数据库进行单独升级。
- **执行步骤**：运行 `kubexm upgrade etcd --cluster=mycluster --to-version=3.5.15`。
- **预期结果**：在不停止集群对外服务的前提下，etcd 成员依次完成镜像更换和重启。

### TC-MAINT-03: 证书自动续期
- **测试目的**：测试集群证书管理。
- **执行步骤**：运行 `kubexm renew kubernetes-certs --cluster=mycluster`。
- **预期结果**：API Server、Kubelet等涉及的叶子证书时间被延后，并相应刷新服务以加载新证书。

### TC-MAINT-04: 配置动态重载
- **测试目的**：测试目标组件无缝重载新配置的能力。
- **执行步骤**：修改配置文件的容器运行时配置，运行 `kubexm reconfigure cluster --cluster=mycluster --target=runtime`。
- **预期结果**：系统仅下发该节点的 containerd 或 docker 新配置并重启相关服务，不干涉其它组件。

### TC-MAINT-05: 综合健康检查
- **测试目的**：确认集群诊断模块的正确性。
- **执行步骤**：运行 `kubexm health cluster --cluster=mycluster --check=component`。
- **预期结果**：屏幕显示完整的表格状组件状态反馈，明确指出任何异常的系统组件或 Pod 状态。

## 6. 架构规范与异常保护专项 (Architecture & Exception Handling)

### TC-ARCH-01: 错误配置 Schema 防御
- **测试目的**：验证配置文件字段与类型强校验功能（Config 模块）。
- **前置条件**：将 `config.yaml` 内的 `kubernetes_version` 字段故意写为 `1234`（非法格式）或删除必须的 CNI 声明。
- **执行步骤**：运行任何读取配置的命令（如 `kubexm create cluster`）。
- **预期结果**：命令在最开始的 Pipeline 预处理阶段因 Schema 校验失败退出，明确报错非法字段，且未连接任何节点产生脏操作。

### TC-ARCH-02: SSH 连接断开与重试 (Connector)
- **测试目的**：验证底层执行引擎的高可用设计。
- **前置条件**：部署过程中，通过防火墙规则使得测试网络出现 15 秒的阻断（模拟网络抖动）。
- **执行步骤**：在阻断期间发起 `kubexm scale cluster ...`。
- **预期结果**：日志显示 `SSH Timeout, Retrying...`。在重试次数未用尽前，网络恢复后，Pipeline 将从中断的 Task/Step 处继续执行或成功抛出错误而非死锁。

### TC-ARCH-03: Pipeline - Task - Step 流转机制
- **测试目的**：确认执行过程符合代码设定的任务层级关系。
- **执行步骤**：随意挑选复杂命令执行（如升级或部署），并收集执行日志。
- **预期结果**：日志具有清晰的前缀结构，展示 `[Pipeline] -> [Task] -> [Step]` 的层次流转。任何一个 Step 执行抛错时，能将上下文传递至上一层，按规定回退或中止。
