# KubeXM 核心架构防守与全参数执行测试用例 (Exhaustive Test Cases)

本文档映射了 KubeXM 代码库中的 17+ 种管道（`internal/pipeline/*`）及所有的命令行控制参数位，设计了对应的验证用例。测试中包含明确的架构底线验证（Config Schema、Connector重试）。

---

## 模块一：架构基石与异常拦截防守 (Architecture & Defense)

### TC-ARCH-SCHEMA-01: YAML Config Schema 零容忍拦截
- **测试目的**：验证底层解析器能否在执行实质动作前识别并阻断任何语法/类型错误。
- **前置条件**：在 `config.yaml` 内，设置不支持的字段（如 `runtime: unknown-runtime`）或将布尔类型错写成字符串（如 `enable: "yes"`）。
- **执行命令**：运行任何基于集群的命令，例如 `kubexm create cluster --cluster=mycluster`。
- **预期结果**：工具在最开始预处理时加载 `internal/config/validator/schema.sh`，随即控制台抛出严格的 Schema 解析失败错误并立即安全中止程序，无任何对主机的 SSH 探针连接行为。

### TC-ARCH-CONN-01: Connector 弱网抗断自愈与层级溯源
- **测试目的**：验证连接引擎 (Connector) 与任务引擎 (Task/Step) 的重试超时机制及规范化溯源。
- **前置条件**：在一次耗时的安装原子步骤中，通过宿主 `iptables` 或断开网卡人为中断跳板机至控制节点的 SSH 连接达 20 秒。
- **执行过程**：监控 `kubexm` 控制台。
- **预期结果**：执行进程被挂起但绝不崩溃闪退。控制台输出重试倒数（`retrying in X seconds...`）。若倒数期内网络恢复，流水线可自动延续；若超时上限触发，则强制按照规范化格式逐级报错：`[Task] 安装组件失败 -> [Step] 下载某文件失败 -> 超时退出`。

## 模块二：离线资产打包及清单 (Download & Manifests)

### TC-ASSET-MANIFEST-01: 跨架构清单演算
- **执行参数**：`kubexm create manifests --kubernetes-version=v1.34.2 --kubernetes-type=kubeadm --arch=arm64 --cni=cilium`
- **预期结果**：控制台输出的组件清单严格匹配传入的矩阵，如拉取的镜像 tags 应为 arm64 架构、版本强校验为 1.34.2 对应的 cilium及kubeadm镜像集。

### TC-ASSET-DOWNLOAD-01: 参数强覆盖与 BOM 完整性测试
- **执行参数**：`kubexm download --cluster=mycluster --kubernetes-version=v1.27.2 --container-runtime=docker`
- **预期结果**：下载引擎忽视配置文件中旧的版本声明，转而按命令行指定的参数将目标包全部缓存，且目录下生成的 `os-bom.txt`, `binary-bom.txt`, `image-bom.txt` 内容与下载实体完全对齐。

### TC-ASSET-ISO-01: 离线构建 ISO 参数轮转
- **执行参数**：分别执行：
  1. `kubexm create iso --with-build-os=ubuntu22 --with-build-arch=amd64`
  2. `kubexm create iso --with-build-all`
  3. `kubexm create iso --with-build-local`
- **预期结果**：调用对应 OS 的构建容器，成功生成带引导、附带 `install.sh` 脚本并适配了架构指令集的安装 ISO。

## 模块三：镜像私库建构流转 (Registry Lifecycle)

### TC-REG-LIFECYCLE-01: 私有库全自动化装卸
- **执行参数**：
  1. `kubexm create registry --cluster=mycluster`
  2. `kubexm delete registry --cluster=mycluster --force --delete-images`
- **预期结果**：精准匹配宿主机 `registry` 角色拉起服务提供 HTTP 端点；强制清理（参数 `--force`）不需二次确认，参数 `--delete-images` 使得 Registry 存储的镜像实体连带挂载目录被连根拔起。

### TC-REG-PUSH-01: 高并发与双向多 Tag 推送验证
- **前置条件**：准备数 GB 镜像包于 packages 中。
- **执行参数**：`kubexm push images --cluster=mycluster --packages --packages-dir=packages/images --parallel=15 --dual --list=/path/to/custom-list.txt`
- **预期结果**：
  - 测试通过参数限流的 `--parallel=15` 高并发工作流内存消耗平稳。
  - `--dual` 参数保证镜像入库时产生双份，既含有原生厂域 Tag (如 `quay.io/xxx`) 又有本地改写 Tag (如 `localhost:5000/xxx`)。
  - 测试使用指定的 `--list` 与 `--packages-dir` 提取镜像源是否精准。

## 模块四：K8s 集群生成与销毁主线 (Cluster Lifecycle)

### TC-CLUS-CREATE-01: 在线公网极简模式 vs. 离线 Air-Gap 物理隔离模式
- **在线执行**：使用单节点公网环境运行 `kubexm create cluster --cluster=mycluster`。
- **离线执行**：在切断外网的 3 节点局域网内，依托此前验证通过的 local registry 与 local packages 运行同样的创建命令。
- **预期结果**：
  - 在线流通过下载管道自动配网拉起。
  - 离线流绝对不可进行任何外网域名解析和下载重试请求，单纯依靠局域网源 100% 拉平所有组件至 `Ready` 状态。

### TC-CLUS-DELETE-01: 集群强制销毁及灾难备份连带
- **执行参数**：`kubexm delete cluster --cluster=mycluster --force --backup=/opt/kubexm-disaster.db`
- **预期结果**：工具在解析到 `--backup` 参数时，改变销毁流水线，第一优先执行 etcd 全量数据快照并保存至 `/opt`，确保存档完毕后再暴力卸载核心组件。

## 模块五：集群弹性扩容及漂移 (Scale & Drain)

### TC-CLUS-SCALE-01: `scale-in` 驱逐优先与 `scale-out` 快速挂载
- **扩容指令**：`kubexm scale cluster --cluster=mycluster --action=scale-out --role=worker --nodes=worker-4`
- **缩容指令**：`kubexm scale cluster --cluster=mycluster --action=scale-in --role=worker --nodes=worker-4`
- **预期结果**：
  - 扩容阶段只在指定机器发生交互，不对核心管理面产生停机波动。
  - 缩容阶段必须严格监听 K8s 的 API 返回值，优先执行节点的 `drain`（安全驱逐）策略，只有当业务 Pods 被调度完后，方可发生底层包的 remove 动作。

## 模块六：复杂系统运维升级套装 (Ops: Upgrade, Renew, Config, Health)

### TC-OPS-UPGRADE-01: 控制面防跌断滚动升级
- **执行参数**：`kubexm upgrade cluster --cluster=mycluster --to-version=v1.33.0` (及后续的 `upgrade etcd`)
- **预期结果**：升级流程严格验证 `--to-version` 的跨度并进行预检，滚动重启间不允许业务级的大规模流量丢失。

### TC-OPS-RENEW-01: 叶子及根证书按需展期
- **执行参数**：交替运行针对不同层级和节点的子命令：`kubexm renew kubernetes-ca/etcd-ca/kubernetes-certs/etcd-certs --cluster=mycluster`
- **预期结果**：检查相应目录下的 `*.crt` 和 `*.key` 生成时间；验证相应 API Server 组件自动重新挂载并生效而无需手动干涉。

### TC-OPS-CONFIG-01: 局部靶向组件热重载
- **执行参数**：`kubexm reconfigure cluster --cluster=mycluster --target=runtime/etcd/cni/lb/addons/hosts`
- **预期结果**：修改配置后只下发对应目标群（如单独更新 containerd 的仓库镜像加速或重启 LB），彻底避免全局重试。

### TC-OPS-HEALTH-01: 集群体检深度剖析表
- **执行参数**：`kubexm health cluster --cluster=mycluster --check=node/component/workload/all`
- **预期结果**：通过不同的 `--check` 探针深入操作系统进程层、容器组件层及工作负载层，生成一份详实清晰、无乱码的表格矩阵输出。

### TC-OPS-RESTORE-01: 底层核心数据单点恢复
- **执行参数**：`kubexm restore cluster --cluster=mycluster --path=/opt/safe_etcd.db`
- **预期结果**：制造大面积节点数据损坏后，仅依靠预设路径的有效 etcd db 文件可强行重启所有状态并召回先前的 Deployments。
