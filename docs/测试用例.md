# KubeXM 详尽测试用例设计 (Detailed Test Cases)

本文档包含了针对 KubeXM 每一个核心模块及指令的极详尽的执行用例设计，涵盖了正常流（Happy Path）、异常边界以及底层架构的专项测试验证。每个用例规定了严格的前置条件、执行步骤和事后校验标准。

---

## 1. 基础命令验证模块 (Base Validation)

### TC-BASE-01: CLI 入口及帮助信息校验
- **测试目的**：确保 CLI 解析器对无参及 `-h` 的处理正确。
- **前置条件**：KubeXM 二进制放入系统的 `$PATH` 且具备可执行权限。
- **执行命令**：
  1. `kubexm`
  2. `kubexm -h`
  3. `kubexm --help`
  4. `kubexm version`
- **预期结果**：
  - 前三条命令应输出格式统一、缩进整齐的帮助清单，涵盖所有核心命令 (`download`, `create`, `delete`, `scale`, `upgrade` 等)。
  - `version` 命令输出类似 `kubexm version x.y.z`。
- **边缘用例**：输入 `kubexm invalid-cmd`，工具应明确报错“未知命令”并退出，返回码非 0。

### TC-BASE-02: 资源清单生成 (Create Manifests)
- **测试目的**：验证工具是否能基于不同的配置生成正确的 BOM 预览。
- **执行命令**：
  1. `kubexm create manifests`
  2. `kubexm create manifests --kubernetes-version=v1.34.2`
  3. `kubexm create manifests --kubernetes-type=kubexm --cni=cilium`
  4. `kubexm create manifests --cluster=test-cluster`
- **预期结果**：
  - 在终端正确打印出目标版本所需下载的镜像名（含 tag）、二进制文件下载 URL 列表及操作系统依赖包名。
  - 读取指定配置 (`--cluster`) 时，输出应当与 `conf/clusters/test-cluster/config.yaml` 声明的一致。

## 2. 离线资源处理模块 (Assets Download & Package)

### TC-DL-01: 标准在线资源下载
- **测试目的**：验证正常配置文件驱动下的全量资源下载。
- **前置条件**：`conf/clusters/mycluster/config.yaml` 存在；机器能正常连接 Docker Hub, GitHub 等公网。
- **执行命令**：`kubexm download --cluster=mycluster`
- **预期结果**：
  - 工具在控制台展示清晰的下载进度条（或并行任务状态）。
  - 下载完成后，检查 `.kubexm/downloads/` 或设定的缓存目录下，包含了对应的 OS, Image, Binary, Helm 包文件。
  - 存在完整的 `*.bom.txt`。

### TC-DL-02: 指定运行时的参数覆盖下载
- **测试目的**：验证 CLI 参数的优先级高于 YAML 配置文件。
- **前置条件**：同上，假设 YAML 中指定运行时为 `containerd`。
- **执行命令**：`kubexm download --cluster=mycluster --kubernetes-version=v1.27.2 --container-runtime=docker`
- **预期结果**：
  - 下载的二进制包中应包含 `docker` 及其相关的 cli 工具，而非 `containerd`。
  - K8s 对应的基础镜像版本应精确指向 v1.27.2。

### TC-DL-03: ISO 离线镜像构建
- **测试目的**：验证使用 Docker 构建特定 OS 的本地源 ISO。
- **前置条件**：系统安装且启动了 Docker。
- **执行命令**：`kubexm create iso --with-build-os=rocky9 --with-build-arch=amd64`
- **预期结果**：
  - 启动 Docker 容器进行源文件拉取与打包。
  - 最终在当前目录或输出目录生成 `*.iso` 文件。
- **事后校验**：将生成的 ISO 挂载至任意目录下，检查其中是否包含了 repodata 目录及预期的 `install.sh`。

## 3. 注册表与离线镜像管理 (Registry & Image Push)

### TC-REG-01: 离线环境搭建私有 Registry
- **测试目的**：验证一键启动私有 Registry 功能。
- **前置条件**：`host.yaml` 指定了一台机器作为 `registry` 角色；`config.yaml` 开启了 registry (`spec.registry.enable=true`)。
- **执行命令**：`kubexm create registry --cluster=mycluster`
- **预期结果**：
  - 工具通过 SSH 登录指定节点，安装并拉起 Docker Registry 容器或原生服务。
  - 本地执行 `curl http://<registry-ip>:5000/v2/` 能够得到 200 返回。

### TC-REG-02: 高并发镜像推送 (Push Images)
- **测试目的**：测试内置的并发机制及重命名机制。
- **前置条件**：已通过 download 获取镜像离线包位于 `packages/images/`，Registry 服务存活。
- **执行命令**：`kubexm push images --cluster=mycluster --packages --parallel=8 --dual`
- **预期结果**：
  - 利用 8 个并发线程将镜像推向私有 Registry。
  - `--dual` 参数使得每个镜像既保留原有前缀（如 `docker.io/...`），同时也被 tag 为私有地址前缀并推送。
  - 查阅 Registry 的 Catalog，确认全量镜像推送无一遗漏。

### TC-REG-03: Registry 安全删除
- **测试目的**：测试一键清理机制。
- **执行命令**：`kubexm delete registry --cluster=mycluster --force --delete-images`
- **预期结果**：跳过终端人工交互，直接关闭 Registry，清理相关配置，并强制删除存储在宿主机上的镜像数据挂载点，确保不留痕迹。

## 4. 核心集群生命周期模块 (Cluster Lifecycle: Create/Delete)

### TC-LC-01: 单机版在线快速部署
- **测试目的**：测试针对最简配置（即开即用）的环境。
- **前置条件**：单台测试机具有 root 权限和外部网络。
- **执行命令**：`kubexm create cluster --cluster=mycluster` (使用最简 config)
- **预期结果**：
  - 自动执行系统参数优化、网络工具下载、kubelet/kubeadm 部署及拉起。
  - 执行 `kubectl get nodes`，当前单机节点处于 `Ready`，所有系统 Pod（如 CoreDNS, kube-proxy, CNI）均处于 `Running` 状态。

### TC-LC-02: 离线多节点全要素部署
- **测试目的**：检验离线环境下利用预下载的 package 和私有 registry 完成复杂拓扑部署。
- **前置条件**：无公网访问；1 个 Master，2 个 Worker，及 1 个 Registry 节点；配置文件正确指向本地 Registry。
- **执行命令**：在跳板机执行 `kubexm create cluster --cluster=mycluster`
- **预期结果**：
  - 集群节点间能够成功通过私有地址互相拉取镜像。
  - 主节点成功初始化，两台工作节点成功 Join 集群。
  - `kubectl get nodes` 列出所有 3 个节点，且状态皆为 `Ready`。

### TC-LC-03: 带前置快照保护的集群销毁
- **测试目的**：验证集群卸载动作以及灾备强制要求。
- **执行命令**：`kubexm delete cluster --cluster=mycluster --backup=/tmp/kubexm-backup.db`
- **预期结果**：
  - 终端首先提示正在执行 etcd 数据备份，检查 `/tmp` 发现备份文件合法生成。
  - 随后按顺序清理所有节点上的容器、挂载目录（`/var/lib/kubelet`, `/etc/kubernetes` 等）及网络规则。
  - 重启主机后执行 `kubectl` 应返回命令未找到或无法连接，确认清理干净。

## 5. 集群弹性扩缩容 (Scale in/out)

### TC-SCALE-01: 工作节点平滑扩容
- **测试目的**：验证单个或多个节点的动态添加。
- **执行命令**：`kubexm scale cluster --cluster=mycluster --action=scale-out --role=worker --nodes=worker-node-3`
- **预期结果**：
  - 仅针对新节点触发初始化 Pipeline（不影响存量节点）。
  - 执行完成后，`kubectl get nodes` 能够看到 `worker-node-3`。

### TC-SCALE-02: 节点安全下线与驱逐
- **测试目的**：验证业务低峰期的缩容操作及安全性。
- **执行命令**：`kubexm scale cluster --cluster=mycluster --action=scale-in --nodes=worker-node-3`
- **预期结果**：
  - 工具先调用 K8s API 对节点执行 `drain` 操作（驱逐该节点上的所有业务 Pod）。
  - 待驱逐完毕，将其从集群剔除。
  - 执行物理清理操作。
- **边缘用例**：若驱逐由于 PDB（PodDisruptionBudget）被卡住，测试工具是否具备超时及报错机制。

## 6. 系统升级与维护 (Upgrade, Renew, Reconfigure)

### TC-MAINT-01: K8s 原地升级测试
- **测试目的**：保证版本跨度升级期间业务的连续性。
- **前置条件**：集群当前版本为 v1.32.4。
- **执行命令**：`kubexm upgrade cluster --cluster=mycluster --to-version=v1.33.0`
- **预期结果**：
  - 执行预检查，确认目标版本包存在。
  - 先升级 Control Plane，随后滚动升级 Worker。
  - 检查版本确认为 v1.33.0 且 CNI 网络保持互通。

### TC-MAINT-02: 证书全量及部分续期
- **测试目的**：验证防止集群证书过期导致不可用。
- **执行命令**：分别执行 `kubexm renew kubernetes-ca --cluster=mycluster` 与 `kubexm renew kubernetes-certs --cluster=mycluster`。
- **预期结果**：
  - `/etc/kubernetes/pki/` 下的对应证书文件时间戳被更新。
  - API Server 与 Controller Manager 读取新证书并重新生效。

### TC-MAINT-03: 指定组件的动态配置重载
- **测试目的**：验证无需全盘重装仅更新部分组件配置的能力。
- **执行步骤**：修改 `config.yaml` 中的 CNI 设置（如修改 Calico IP Pool）。
- **执行命令**：`kubexm reconfigure cluster --cluster=mycluster --target=cni`
- **预期结果**：仅下发并应用 CNI 相关的新配置，不会影响到容器运行时或 etcd。

### TC-MAINT-04: 数据备份与灾备恢复
- **测试目的**：验证底层 etcd 快照机制。
- **执行命令**：
  1. 写入一条测试用的 K8s ConfigMap。
  2. `kubexm backup cluster --cluster=mycluster --path=/tmp/backup`。
  3. 删除该 ConfigMap 并在任意节点破坏 etcd 进程。
  4. `kubexm restore cluster --cluster=mycluster --path=/tmp/backup`。
- **预期结果**：恢复命令执行后，集群恢复正常健康状态，此前写入的 ConfigMap 依然存在。

## 7. 架构级异常保护验证 (Architecture, Validation & Robustness)

### TC-ARCH-01: YAML Schema 严格校验拦截
- **测试目的**：验证错误配置不能进入执行层。
- **执行步骤**：修改 `config.yaml`，填入不支持的容器运行时类型 `runtime: unknown-engine`。
- **预期结果**：解析器应在秒级返回详细报错（类似 "Invalid value unknown-engine for runtime"），Pipeline 立刻中止，不发生任何 SSH 连接。

### TC-ARCH-02: 连接引擎 (Connector) 超时容错
- **测试目的**：验证网络剧烈波动时的防御机制。
- **执行步骤**：在部署核心流程时，使用 `iptables` 丢弃跳板机到某 Worker 的 22 端口流量，持续 30 秒。
- **预期结果**：
  - SSH 连接器不会死锁，日志打印 "Connection Timeout, retrying...".
  - 网络恢复后流程继续推进。若超时超过预设阈值（如 3 次），则抛出连接致命错误并中止该节点的部署。

### TC-ARCH-03: 架构规范与层级日志
- **测试目的**：验证 `Pipeline -> Task -> Step` 代码层级被严格遵守。
- **预期结果**：人工审查控制台输出，确认包含 `[Pipeline]`, `[Task]`, `[Step]` 标签的结构化输出；任何执行出错必须精确定位至具体的 `[Step]` 名称及其所执行的具体 Bash 片段。
