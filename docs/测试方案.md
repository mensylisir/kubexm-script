# KubeXM 全极限界深度测试方案 (Exhaustive Test Strategy)

## 1. 方案目标与架构对齐

KubeXM 是一个高度严谨的、通过 `Pipeline -> Task -> Step` 代码架构编排的 Kubernetes 集群全生命周期管理与分发部署平台。
为确保工具在极端复杂的环境与无限可能的配置组合下仍能保持绝对的健壮性，本测试方案将采用**“全极限界组合穷举法”**，要求实现 100% 命令分支覆盖、100% 选项位覆盖（Option flags）、全量 OS 及网络架构覆盖，并严防破坏架构规范。

### 1.1 核心测试对象
完全覆盖 `bin/kubexm` 以及内部 `internal/pipeline/` 中暴露的所有主命令与子命令：
`download`, `create cluster/registry/manifests/iso`, `delete cluster/registry`, `push images`, `scale cluster`, `upgrade cluster/etcd`, `renew kubernetes-ca/etcd-ca/kubernetes-certs/etcd-certs`, `backup cluster`, `restore cluster`, `health cluster`, `reconfigure cluster`。

## 2. 穷举参数矩阵与条件分支策略

测试方案必须围绕以下极度细化的多维条件展开：

### 2.1 底层执行环境分支 (Environment Branches)
- **物理架构**：单节点 (All-in-one)、标准高可用 (3M+3W)、跨网段异构集群。
- **网络条件**：
  - **Online**：具有无限制的外部网络、DNS 和 docker.io 访问权限。
  - **Air-Gap (纯离线)**：切断物理网卡，仅允许通过内置的 `packages` 和 Local ISO、私有 Registry 进行包的分发与部署。
- **证书与鉴权分支 (Auth/TLS)**：使用 `--cacert`, `--cert`, `--key` 针对远程外部组件（如 external etcd, external Registry API）进行严苛的 TLS 双向认证通讯。

### 2.2 CLI 与配置属性全量覆盖矩阵 (Parameter Matrix)
所有命令在测试时必须组合覆盖代码库中识别到的每一个 flag（包含隐藏或高级配置）：
- **全生命周期控制流**：`--cluster`, `--action` (scale-out/scale-in), `--role`, `--nodes`, `--to-version`。
- **交互与拦截绕过**：模拟终端交互中的“Y/N”按钮逻辑，测试 `--force` 参数对各类危险指令（`delete`, `scale-in` 驱逐）的安全绕过及静默处理。
- **镜像与包管理流**：`--packages`, `--packages-dir`, `--parallel` (并发上限), `--dual` (双重标签), `--list`, `--manifest` (Docker 多架构索引清单推送), `--target-registry` (镜像远端投递), `--delete-images`。
- **集群组件定义流**：`--kubernetes-version`, `--kubernetes-type` (kubeadm vs kubexm), `--container-runtime` (docker, containerd, crio, podman), `--cni` (calico, flannel, cilium), `--arch` (amd64, arm64)。
- **ISO 自举编译流**：`--with-build-os` (centos7, rocky9, ubuntu22 等), `--with-build-os-version`, `--with-build-arch`, `--with-build-all`, `--with-build-local`。
- **高阶运维探针流**：`--check` (node, component, workload, all), `--target` (runtime, etcd, cni, lb, addons, hosts), `--path` (快照路径), `--endpoints` (外部接入点), `--output-format` (如 json, table)。

## 3. 架构底线防守与异常注入策略 (Architecture Boundaries)

### 3.1 Pipeline / Config Schema 强制拦截
- **配置免疫**：对 YAML 的 Schema 进行破坏性构造（例如字段格式错乱、引用不存在的 LB 类型）。`internal/config/validator/schema.sh` 必须展现“零容忍”态度，在 Pipeline 发起第 1 步前立刻中断并打印校验堆栈，禁止向节点发送任何越权指令。
- **配置幂等**：任何 `create` 和 `reconfigure` 命令必须具有绝对的幂等性设计，执行 10 次与执行 1 次对底层文件系统的影响必须等同，禁止雪崩与重复挂载。

### 3.2 Connector 断流与自愈溯源机制
- **网络震荡测试**：使用底层系统工具（如 `tc`）在拉取镜像、配置容器运行时等任意原子 `Step` 阶段注入随机丢包或 22 端口阻断。
- **自愈要求**：Connector 不能引发上层 bash 的无限僵死（Deadlock）。必须触发超时轮询。
- **追溯要求**：重试耗尽后，报错必须严格遵循 `[Pipeline] -> [Task] -> [Step] -> Timeout` 的堆栈打印，符合架构日志规范。

## 4. 质量签署标准 (Sign-off Criteria)
所有的测试矩阵用例不仅要关注功能“能不能用”，还必须审查系统级调用的准确度，并在《测试用例.md》中全部体现。仅在全参数无一遗漏且所有 P0/P1 防守成功的前提下允许发布。
