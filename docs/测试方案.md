# KubeXM 测试方案

## 1. 测试背景与目的
KubeXM 是一个采用模块化架构设计的 Kubernetes 集群部署和管理工具，支持离线下载、单节点及多节点集群创建与维护。
本测试方案旨在定义明确的测试策略，确保工具核心功能的正确性、各种执行分支的稳定性以及架构规范的严格性，从而保证在各种复杂环境下部署和维护 Kubernetes 集群的可靠性。

## 2. 测试目标
- **核心功能正确性**：验证 `download`, `create`, `delete`, `scale`, `upgrade`, `renew`, `backup`, `restore`, `health`, `reconfigure`, `push images` 等所有支持的命令均能达到预期效果。
- **配置与参数覆盖**：验证在用户传入不同参数（如 `--kubernetes-version`, `--container-runtime`, `--cni`）或修改配置文件 (`config.yaml`/`host.yaml`) 的场景下，工具能否正确处理各类分支逻辑。
- **架构一致性与健壮性**：确保系统执行严格遵循 `Pipeline -> Task -> Step` 的分层架构，遇到网络异常、配置不合规、权限不足等情况时，能够通过 `config` 校验模块和 `connector`（SSH 执行引擎）进行合理的拦截和优雅的异常处理。
- **业务场景覆盖**：完全覆盖在线部署和离线部署两套独立的核心应用场景。

## 3. 测试范围
- **基础与辅助命令**：版本查看、帮助信息及生成资源清单 (`manifests`)。
- **离线资源管理与BOM构建**：ISO 离线镜像构建、各类资源下载 (`binary`, `image`, `os`, `helm`) 及对应的 BOM 文件一致性。
- **核心生命周期**：单节点集群部署、多节点集群部署、注册表 (`Registry`) 的创建与管理、镜像推送、集群卸载及节点扩缩容。
- **维护管理**：K8s 及 etcd 版本升级、CA及各组件证书续期、数据备份与恢复、组件配置重载以及集群健康状态检查。
- **架构防错能力**：YAML 解析器 schema 检查拦截、网络连接故障重试以及依赖包校验。

## 4. 测试环境要求
为尽可能模拟真实使用场景，需要在不同类型的环境上进行测试：

### 4.1 操作系统支持 (OS)
需要在 RPM 系和 DEB 系的代表系统中进行覆盖：
- **RPM系**：Rocky Linux 9 或 CentOS 7
- **DEB系**：Ubuntu 22.04

### 4.2 网络隔离要求
- **在线环境**：能够直接访问外部网络，用于验证在线集群部署、资源下载（下载镜像、Helm Charts 和二进制文件）。
- **离线环境**：物理隔离或安全组阻断外部网络连接，仅能使用已下载的资源包 (`packages/`) 或通过 ISO 安装环境，用以验证离线安装、注册表创建与本地镜像推送。

### 4.3 集群架构与节点要求
- **环境A（单节点测试）**：1台服务器，用于测试命令的基础连通性和单节点快速创建功能。
- **环境B（多节点测试）**：3台以上服务器 (例如 1 Master + 2 Worker 甚至独立 etcd/registry 节点)，用于验证多角色配置和集群缩扩容操作 (`scale-out`/`scale-in`)。
