# KubeXM 全极限界深度测试方案 (Exhaustive Test Strategy)

## 1. 方案目标与架构对齐

KubeXM 是一个高度严谨的 Kubernetes 集群全生命周期管理与分发部署平台，完全基于 `Pipeline -> Task -> Step` 代码架构编排。
为确保工具在极端复杂的环境与无限可能的配置组合下仍能保持绝对的健壮性，本方案要求实现 100% 命令分支覆盖、100% 选项位参数（Option flags）穷举、全量操作系统架构矩阵覆盖，并严防破坏架构规范。

### 1.1 核心测试对象
完全覆盖 CLI 暴露的所有主命令与子命令：
`download`, `create cluster/registry/manifests/iso`, `delete cluster/registry`, `push images`, `scale cluster`, `upgrade cluster/etcd`, `renew kubernetes-ca/etcd-ca/kubernetes-certs/etcd-certs`, `backup cluster`, `restore cluster`, `health cluster`, `reconfigure cluster`。

## 2. 穷举参数矩阵与底层核心流转策略

### 2.1 底层执行环境分支 (Environment Branches)
- **网络条件流转**：
  - **Online**：具备外网访问，验证原生组件直连下载路径。
  - **Air-Gap (离线)**：切断网卡，验证严格基于 `os-bom`, `image-bom`, `binary-bom` 约束的本地依赖分发，阻拦任何公网 `curl` 请求。

### 2.2 CLI 全量选项矩阵覆盖 (Parameters Matrix)
所有的执行路径均须在测试用例中交叉使用以下全部标识（Flags）：
- `--action`, `--arch`, `--backup`, `--cacert`, `--cert`, `--check`, `--cluster`, `--cni`
- `--container-runtime`, `--delete-images`, `--dual`, `--endpoints`, `--force`, `--help`
- `--key`, `--kubernetes-type`, `--kubernetes-version`, `--list`, `--manifest`, `--nodes`
- `--output-format`, `--packages`, `--packages-dir`, `--parallel`, `--path`, `--role`
- `--target`, `--target-registry`, `--to-version`, `--version`
- `--with-build-all`, `--with-build-arch`, `--with-build-local`, `--with-build-os`, `--with-build-os-version`

### 2.3 核心底层模块交互防守 (Internal Step Boundaries)
针对已经探明的核心执行路径（如 `certs/common`, `certs/etcd`, `certs/kubeadm`, `cni` 等模块），必须严查其 API 交互及组件隔离逻辑：
- **`certs/kubeadm` 验证**：在执行 `renew kubernetes-ca` 时，验证从 `rotate_collect` 到 `distribute_pki_k8s_copy` 的层级调用，确保新证书能精确分发至所有节点的 `/etc/kubernetes/pki` 并修复所有权。
- **`certs/etcd` 验证**：在执行 etcd 证书轮调时，验证 `renew_etcd_ca` 及客户端对接参数 `--endpoints`, `--cacert`, `--cert`, `--key` 的合法连通性，模拟双向 TLS 安全准入失败阻断机制。
- **`cni` 网络插件验证**：配置不同 `--cni` (calico, cilium, flannel) 部署后，测试 CNI 组件的重置功能（例如执行卸载集群时验证 `calico_delete.sh` 等专项清除脚本能否干净回收网络网桥）。

## 3. 架构底线与按钮逻辑的 CLI 映射

### 3.1 拦截与 Bypass (“强制”按钮模拟)
在交互式界面中的“确认/绕过”按钮，在 CLI 中被抽象为 `--force` 选项。
- **危险绕过验证**：执行 `delete cluster` 及 `scale-in` 等存在高风险退出的命令，必须拦截报错。只有附加 `--force` 才能等价于触发“高危风险同意”的逻辑开关，绕过询问并执行底层拆包动作。

### 3.2 Config Schema 与 API 准入免疫
- 对 YAML Config Schema 构造无效属性，系统必须利用 `validator` 拦截执行，绝对不准启动内部 Pipeline。

## 4. 质量签署标准 (Sign-off Criteria)
所有的参数矩阵（近 35 个参数的笛卡尔积边界）与底层 Step 防御机制（Certs, CNI 清理等）的反馈均必须在《测试用例.md》中全部体现。仅在全参数无遗漏且所有防守（如证书隔离失败阻断）成功通过验证的前提下允许签发“可发布”许可。
