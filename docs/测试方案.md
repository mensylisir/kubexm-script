# KubeXM 全极限界深度测试方案 (Exhaustive Test Strategy)

## 1. 方案目标与架构对齐
KubeXM 是一个基于 `Pipeline -> Task -> Step` 代码架构编排的 Kubernetes 集群全生命周期管理与分发部署平台。
本测试方案的终极目标是：提供一套绝对百科全书式的、能够验证其在所有可能的配置矩阵、网络架构、以及全部 17+ 种指令和 30+ 项参数选项标志下，仍能稳健运行的策略。

### 1.1 全量指令测试边界
必须对 `bin/kubexm` 中所有的命令进行全方位压测：
- **资源与引导篇**：`download`, `create manifests`, `create iso`。
- **镜像仓库篇**：`create registry`, `push images`, `delete registry`。
- **生命周期篇**：`create cluster`, `delete cluster`, `scale cluster`。
- **运维与灾备篇**：`upgrade cluster/etcd`, `renew kubernetes-ca/etcd-ca/kubernetes-certs/etcd-certs`, `backup cluster`, `restore cluster`, `health cluster`, `reconfigure cluster`。

## 2. 核心测试策略与架构防守

### 2.1 架构底座防守验证 (Architecture Defense)
- **配置免疫 (Schema Validation)**：在解析 `config.yaml` 阶段，注入各类错乱数据，系统预处理器（位于 `internal/config/validator`）必须体现出“零容忍”，即刻报错并阻断，杜绝产生任何脏调用。
- **网络震荡自愈 (Connector Resilience)**：模拟恶劣网络（如跳板机到节点间丢包）。Connector 层不允许导致执行挂起死锁，必须基于设定的超时容忍度轮询重试。重试耗尽后必须输出规范的 `[Pipeline]->[Task]->[Step]` 报错调用栈。
- **原子操作可追踪性 (Step Boundaries)**：在涉及 `internal/step/certs/*`、`internal/step/cni/*_delete` 脚本的具体流转中，验证错误被精准捕获，绝不发生静默失败。

### 2.2 CLI 标志控制位 (Flags) 极限界映射
所有功能不仅要测主线，还需深入验证命令行标志的生效与覆盖逻辑：
- **交互与绕过策略**：将 `--force` 选项视为核心的风险控制门闩。在针对 `delete cluster`, `delete registry`, 或存在强行驱逐风险的操作时，验证是否正确绕过了人工干预确认对话。
- **环境接驳与外置 API 策略**：验证与独立外部存储通信时，利用 `--endpoints`, `--cacert`, `--cert`, `--key` 构建出合格且阻断异常篡改的双向 TLS 连接。
- **流转重塑策略**：带 `--backup` 参数时，必须验证流水线优先级被重置（即在销毁集群网络和核心服务前，必须强行触发数据下盘）。

### 2.3 网络物理隔离验证 (Air-Gap)
设定“Online(全网接入)”与“Air-Gap(纯局域物理隔离)”两条截然不同的测试通道：
- **离线强制要求**：使用离线模式（依赖 `packages` 目录）时，所有相关子模块（含 `internal/step/download/*`）必须禁止任何形式的域名寻址与下载外联。

## 3. 质量发布底线 (Sign-off Criteria)
所有的参数笛卡尔积组合测试，以及对架构边界发起的破坏性网络/配置测试，必须得到 100% 通过（即表现符合架构设计预期的回退或中止），方可签发生产可用证明。
