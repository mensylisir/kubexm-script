# KubeXM 深度测试方案与策略规范 (Test Strategy & Architecture Norms)

## 1. 编写背景与测试核心目的

KubeXM 是一个高度模块化、通过严格的 `Pipeline -> Task -> Step` 架构编排执行的 Kubernetes 集群部署和管理工具。为了保证工具在极其复杂的组合环境（在线网络/纯物理隔离、几十种操作系统、多种 CNI 插件及运行时容器组合）下均能健壮且符合预期的运行，特编制此极度详尽的测试方案。

本方案的核心目的：
1. **全命令与全参数覆盖**：囊括 `kubexm` 支持的 17+ 个子命令及其多达几十项的参数矩阵组合。
2. **执行分支与条件组合**：涵盖在线部署、离线离地部署（配合私有 Registry 和本地 ISO）、单节点、集群架构、扩缩容状态的条件分流。
3. **架构规范保真 (Architecture Compliance)**：强制要求测试配置的强校验（Schema 验证）、网络连接的健壮性容错（Connector 重试机制）及原子操作步骤隔离（Step 机制）。

---

## 2. 全量测试对象与参数矩阵分析

为了保证没有任何分支被遗漏，测试方案将所有的执行变量分解为“测试矩阵”。

### 2.1 核心主干命令树与参数策略
- **`download` 命令**
  - **核心测试点**：包管理的 BOM 生成（Binary, Image, OS, Helm）。
  - **参数覆盖**：`--cluster` (依据 config 提取配置), `--kubernetes-version` (强覆盖 YAML 版本), `--container-runtime` (覆盖运行时), `--cni` (覆盖 CNI 插件)。
- **`create` 命令组**
  - `create cluster`：在线/离线下载联动部署，参数 `--cluster` 驱动的全流程。
  - `create registry`：通过节点角色及 `spec.registry.enable` 布尔条件分流。
  - `create manifests`：针对版本 (`--kubernetes-version`)、架构 (`--arch` [amd64/arm64])、类型 (`--kubernetes-type` [kubeadm/kubexm]) 的矩阵组合及预览准确性。
  - `create iso`：编译不同操作系统及版本的安装媒介 (`--with-build-os`, `--with-build-arch`, `--with-build-all`, `--with-build-local`)。
- **`delete` 命令组**
  - `delete cluster`：安全检查交互，`--force` 强制条件，`--backup` 连带 etcd 灾备的分支路径。
  - `delete registry`：连带 `--delete-images` 彻底清理条件的分支。
- **`push` 命令组**
  - `push images`：离线私有源管理，覆盖 `--packages` 目录获取、`--list` 指定文件、`--dual` 双重 Tag 模式、`--parallel` 多并发限制。
- **`scale` 命令组**
  - `scale cluster`：方向 (`--action` [scale-out/scale-in])、范围 (`--role` [worker/control-plane] 及 `--nodes` 精确指定) 对应的节点驱逐/初始化。
- **`upgrade` / `renew` 命令组**
  - `upgrade cluster/etcd`：精准测试滚动升级的连续性，参数 `--to-version` 的兼容性拦截。
  - `renew kubernetes-ca/etcd-ca/kubernetes-certs/etcd-certs`：CA 根证书与叶子终端证书的不同更新频率及刷新重启。
- **`backup` / `restore` 命令组**
  - etcd 快照及恢复，核心条件 `--path`。
- **`health` / `reconfigure` 命令组**
  - `health cluster`：不同探针深度 `--check` [all, node, component, workload]。
  - `reconfigure cluster`：动态配置热重载 `--target` [all, runtime, etcd, cni, lb, addons, hosts]。

### 2.2 环境组合条件（配置矩阵）
- **操作系统矩阵 (OS)**：CentOS 7/8/9, Ubuntu 18.04/20.04/22.04, Debian 10/11, RHEL 8/9, Rocky 8/9, AlmaLinux 8/9, Fedora 36-38, UOS, Kylin V10。必须抽样涵盖 RPM 系与 DEB 系的底层包管理差异。
- **K8s 组件矩阵**：
  - 运行时：Docker, containerd, cri-o, Podman。
  - 网络插件 (CNI)：Calico, Flannel, Cilium, Kube-OVN, Hybridnet。
  - 负载均衡 (LB)：HAProxy+Keepalived, Kube-VIP, Nginx。

## 3. 测试策略设计

针对不同维度，采取以下核心测试策略来完成极尽详尽的验证工作：

### 3.1 架构规范与边界防守测试 (Architectural Defense Strategy)
KubeXM 设计的核心是高度可靠性，测试时必须模拟破坏以验证其架构边界：
- **Config Schema 零容忍**：在 YAML (`config.yaml` 或 `host.yaml`) 中引入类型不匹配（例如将数字赋给布尔值）、未声明属性或极端的配置（如不存在的版本号），测试 Pipeline 预处理阶段的 Schema Validator 是否百分之百阻拦，防止产生脏操作。
- **执行引擎容错 (Connector/Runner)**：利用 `iptables` 或 `tc` (Traffic Control) 在命令执行关键步骤（例如 `kubeadm init`）引入丢包和 SSH 闪断，验证 Runner 能够优雅重试直至达到超时上限，之后准确上报具体出错的 `Task` 及 `Step`，而不是进程永久卡死。

### 3.2 离线隔离与包管理策略 (Air-Gap & BOM Verification)
- **绝对断网原则**：在验证离线功能时（使用 pre-downloaded 的 packages），必须关闭所有的 DNS 解析及外网网关，确保 `create cluster` 全流程不能有一字节的数据来自公网，强制验证 BOM (os-bom, image-bom, binary-bom, helm-bom) 描述的完整无缺。

### 3.3 状态机流转与幂等性策略 (Idempotency & State transitions)
- **重入执行测试**：所有配置类命令（如 `reconfigure`、多次执行 `create`），必须能在成功后重复执行而不引起集群雪崩，测试系统是否能在部署前正确判定状态（组件是否已存在、证书是否匹配）。
- **生命周期破坏与自愈**：如强制杀死部分系统核心 Pod (CoreDNS 等) 后执行 `health cluster --check=component`，验证系统探测与报警的精准度。

## 4. 缺陷管理与放行基准 (Defect Management & Sign-off)

1. **Bug 优先级 (P0-P3)**：任何导致集群核心能力瘫痪、配置文件 Schema 穿透导致非预期行为的 Bug 均定性为 P0/P1，绝对禁止上线。
2. **自动化与回归通过率**：全套参数及组合在单元测试 (`make test`，覆盖 81+ 用例及 Docker integration) 下必须达到可用容忍度（除受限沙盒的特定跳过外，关键核心业务不出现 FAIL）。
3. **多维条件矩阵的签署准出**：必须完成至少 3 个差异化操作系统（例：Ubuntu22, Rocky9, CentOS7）、2 种运行时（Docker, Containerd）及在线/离线完整生命周期演练的报告，方案才可标记为“Ready for Production”。
