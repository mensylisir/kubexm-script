# KubeXM 全面测试方案 (Test Strategy)

## 1. 概述 (Overview)

### 1.1 编写目的
本文档旨在为 KubeXM（一个高度模块化、支持离线/在线环境的 Kubernetes 集群部署与生命周期管理工具）制定系统性、全面性、标准化的测试策略。它将作为整个测试活动的指导方针，确保系统发布前的质量，覆盖全部核心命令、参数组合、执行分支及底层架构规范。

### 1.2 测试对象与范围
**测试对象**：KubeXM CLI 工具及底层运行依赖的全部 Shell 模块。
**全量覆盖范围**：
- **核心 CLI 命令**：`download`, `create`, `delete`, `scale`, `upgrade`, `renew`, `backup`, `restore`, `health`, `reconfigure`, `push images` 及其所有的子命令与可选参数。
- **环境架构**：在线公网环境与纯物理隔离的离线环境。
- **目标操作系统 (OS)**：RPM系 (CentOS 7/8/9, RHEL, Rocky 8/9, AlmaLinux 等) 与 DEB系 (Ubuntu 18.04/20.04/22.04, Debian 10/11)。
- **K8s 生态组合**：多种容器运行时 (Docker, containerd, crio)、多款 CNI (Calico, Flannel, Cilium等)、多种 LoadBalancer (HAProxy, Kube-VIP等)。
- **架构与模块规范**：配置加载与校验 (Config)、全流程调度执行引擎 (Pipeline/Task/Step)、SSH 连接器 (Connector) 的健壮性。

## 2. 测试策略 (Test Methodologies)

针对 KubeXM 复杂的分布式执行和跨平台特性，采用以下组合测试策略：

### 2.1 功能测试 (Functional Testing)
- **黑盒测试**：从终端用户的角度，完全依据官方文档执行命令，验证系统的标准输出、退出码及最终在节点侧生效的状态（如 K8s 节点是否 Ready，组件是否 Running）。
- **参数矩阵测试**：对涉及多种组合的命令（如 `download` 与 `create manifests`），通过正交法则穷举关键参数组合（如 `--kubernetes-version` + `--container-runtime` + `--cni`）。

### 2.2 兼容性测试 (Compatibility Testing)
- **跨系统验证**：确保 OS BOM（操作系统依赖包）的精准性，针对 Rocky 9, Ubuntu 22.04 等典型系统分别运行安装测试。
- **高低版本验证**：测试旧版 Kubernetes (如 v1.25) 与新版 (v1.34) 的兼容情况。

### 2.3 异常与容错测试 (Negative & Robustness Testing)
- **输入异常拦截**：提供格式错误的 YAML 配置文件或不合法的参数，验证前置 Schema Validation 的阻断机制。
- **网络与系统异常**：
  - 模拟下载源 (Registry/YUM) 超时或 404，验证工具的失败处理机制。
  - 部署途中模拟目标机 SSH 闪断、端口拒绝或目标盘空间耗尽，验证 Connector 层的重试机制与优雅退出机制。
- **幂等性测试**：针对相同的集群执行多次 `create cluster` 或是相同的 `reconfigure` 命令，验证无破坏性副作用。

### 2.4 架构规范测试 (Architecture Validation)
- 验证代码中 `Pipeline -> Task -> Step` 的流转模式，确保所有日志输出具备标准前缀且层级关系明确。
- 确认未抛出隐式的 Bash 异常，所有 `set -e` 触发的退出均经过统一的 `error.sh` 捕获并输出。

## 3. 测试环境设计 (Test Environment Design)

测试应构建三套标准化的沙箱环境：

### 3.1 环境 A (在线快速单测环境)
- **网络条件**：开启公网代理的极速下载环境。
- **硬件配置**：1台 2C4G 的 Linux 虚拟机（如 Ubuntu 22.04）。
- **测试用途**：单节点快速拉起、语法树检查、`manifests` 验证和轻量级的 CLI 测试。

### 3.2 环境 B (离线多节点全生命周期环境)
- **网络条件**：严格断网（仅保留内网互通的 SSH）。
- **硬件配置**：4台 4C8G 虚拟机（1 控制节点, 2 工作节点, 1 Registry/跳板节点）。
- **测试用途**：测试离线 ISO 挂载、BOM 包分发、内网 Registry 镜像并发推送、多节点高可用拉起以及集群水平扩缩容 (`scale-out/in`)。

### 3.3 环境 C (灾难恢复与维护演练环境)
- **网络条件**：正常联网。
- **硬件配置**：3台 4C8G 虚拟机。
- **测试用途**：etcd 和 K8s 的版本滚动升级测试、etcd 的数据快照备份与恶意破坏后的恢复 (`backup/restore`)、全量证书的过期与强制续期。

## 4. 测试准入与准出标准 (Entry and Exit Criteria)

### 4.1 准入标准 (Entry Criteria)
- 代码通过所有静态检查（Shellcheck），`make test` 单元测试通过率达到 100%。
- 开发输出详细的变更日志 (Changelog)。
- 测试环境的虚拟机快照准备就绪。

### 4.2 准出标准 (Exit Criteria)
- 100% 覆盖本文档及《测试用例.md》定义的核心主流程用例。
- 不存在 P0/P1（致命与严重）级别 Bug，如数据丢失、节点被意外破坏、严重卡死等。
- 已发现的 P2 级 Bug 必须提供可行的 Workaround。
- 离线环境及多节点场景下集群功能一次性验证通过率达到 95% 以上。

## 5. 缺陷管理与流程 (Defect Management Process)
- **缺陷级别定义**：
  - **P0 致命**：核心主命令崩溃、数据损毁、配置引发系统级故障。
  - **P1 严重**：重要功能完全不可用（如离线下载缺失依赖、节点加入失败）、没有规避方案。
  - **P2 中等**：功能部分受限但存在 Workaround，或非核心日志信息丢失。
  - **P3 轻微**：控制台排版、文档错别字、不影响使用的轻度显示异常。
- **流转要求**：测试阶段每天统计 Bug 收敛情况；提交修复前必须确保无回退影响。
