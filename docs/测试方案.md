# KubeXM 全面深度的测试方案 (Test Strategy)

## 1. 测试对象与目标概述

KubeXM 是一个采用严谨的 `Pipeline -> Task -> Step` 代码架构编排、高度模块化的 Kubernetes 集群全生命周期管理工具。
本方案旨在提供一套绝对详尽的测试策略，用于验证其在全矩阵参数组合（涵盖超过 25 种 CLI 标志）、复杂运行环境（涵盖数十种 OS 发行版、各类 CNI、不同容器运行时以及离线隔绝网络）下的鲁棒性，以及它对于系统底线（Schema 校验与 SSH 异常重试）的坚守。

**核心测试目标：**
- 覆盖 17+ 种主干子命令及其庞杂的执行参数（如 `--packages-dir`, `--dual`, `--target`, `--check` 等）。
- 覆盖所有网络分支条件（完全公网在线拉取 vs. 纯物理 Air-Gap 断网下依靠 `packages` 目录）。
- 覆盖架构约束规范，验证任何非法配置和恶劣网络波动不会造成程序的隐蔽崩溃或脏写。

## 2. 测试覆盖范围矩阵 (Test Matrix)

为确保不遗漏任何一条执行分支，测试矩阵从以下三个维度进行穷举：

### 2.1 OS 与 K8s 生态矩阵
| 维度 | 支持范围 | 测试抽样策略 |
| :--- | :--- | :--- |
| **操作系统 (OS)** | CentOS 7-9, RHEL 8-9, Rocky 8-9, AlmaLinux, Ubuntu 18.04-22.04, Debian 10-11, UOS, Kylin | 必须包含：Rocky 9 (RPM代表) 与 Ubuntu 22.04 (DEB代表) 的正向部署及错误拦截测试。 |
| **容器运行时** | Docker, containerd, cri-o, Podman | 必须包含：containerd (默认主推) 与 Docker。 |
| **网络插件 (CNI)** | Calico, Flannel, Cilium, Kube-OVN, Hybridnet | 必须包含：Calico 与 Cilium。 |
| **负载均衡 (LB)** | HAProxy+Keepalived, Kube-VIP, Nginx | 验证 Kube-VIP 与 HAProxy 组合。 |

### 2.2 CLI 命令与参数分支流转
必须对 `bin/kubexm` 中所有的 Parser 入口进行功能验证：
- **资源离线准备流**：`download` (参数覆盖 `--kubernetes-version`, `--cni`), `create manifests`, `create iso` (`--with-build-os`, `--with-build-arch`, `--with-build-all`, `--with-build-local`)。
- **镜像注册表流**：`create registry`, `push images` (`--packages`, `--packages-dir`, `--parallel`, `--dual`, `--list`, `--target-registry`), `delete registry` (`--force`, `--delete-images`)。
- **核心生命周期**：`create cluster` (受 `config.yaml` 驱动的在线/离线分支流转), `delete cluster` (`--force`, `--backup`)。
- **动态伸缩流**：`scale cluster` (`--action`=[scale-out/scale-in], `--role`=[worker/control-plane], `--nodes`)。
- **运维与灾备流**：`upgrade cluster/etcd` (`--to-version`), `renew kubernetes-ca/etcd-ca/kubernetes-certs/etcd-certs`, `backup cluster` / `restore cluster` (`--path`), `health cluster` (`--check`), `reconfigure cluster` (`--target`)。

## 3. 测试策略实施规范 (Methodologies)

### 3.1 架构底线防守策略 (Architecture Defense Validation)
系统对架构原则的坚持是 KubeXM 稳定性的核心，本策略强制进行如下破坏性验证：
- **Config Schema 零容忍**：在执行读取 `config.yaml` 或 `host.yaml` 的任何命令前，注入格式错误（如字段类型混淆）或逻辑错误（如未定义 registry role 却开启 registry=true）。测试框架必须在最外层抛出明确 Schema Error，绝对不能产生实质的 SSH 握手或脏部署操作。
- **Connector (SSH/执行器) 强压与重试**：利用网络工具 `tc` 或 `iptables` 在执行原子 `[Step]` 途中间歇性阻断 22 端口，验证连接器是否按设定的超时时间和次数进行轮询重试。
- **执行层级可追溯性**：代码规范要求 `Pipeline -> Task -> Step`。日志策略必须明确检查控制台输出，发生异常时，能够输出带有具体 `[Step]` 前缀的报错上下文。

### 3.2 离线隔离与 BOM 同源策略 (Air-Gap & BOM)
测试离线执行分支时，必须物理或逻辑断开公网，以此验证：
- 工具不包含任何硬编码的公网地址下载逻辑。
- 所有的资源（OS 依赖包、Image、Binary）必须完全匹配 `download` 阶段生成的对应的 BOM 文本。

### 3.3 并发与安全边界策略 (Concurrency & Safety)
- 测试 `push images --parallel=N` 时的线程控制与内存占用。
- 测试涉及到节点清理和卸载的操作（`delete cluster`, `scale in`）时，前置 `drain` 驱逐 API 的调用是否被正确触发和监控超时；执行删除必须要有 `--force` 或者严密的交互对话拦截。

## 4. 质量要求与缺陷定义 (Quality Gates & Bug Def)
- **P0 致命**：任何违背架构规范（错误配置穿透导致脏动作）、导致节点文件系统或数据不可逆损毁的问题。
- **P1 严重**：全量参数覆盖中，某个 CLI 参数未生效或存在逻辑冲突；离线安装时尝试去公网请求导致阻塞。
- **准出标准**：所有在 `测试用例.md` 中定义的，涵盖上述全部标志位和执行分支的用例，通过率需到达 100%。
